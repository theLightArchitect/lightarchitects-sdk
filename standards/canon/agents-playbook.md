## Part I — Roles and Responsibilities

### 1.1 The agent roster

| Agent | Role | Temperature | Spawned by | Context scope |
|---|---|---|---|---|
| **Governor** | Orchestration, state machine driver, HITL escalation, merge authority | 0.2 | Operator or `/BUILD` meta-skill | Full program + build |
| **Worker** | Implementation, commits, wave-level gates | 0.3 | Governor | Single task + wave |
| **Gatekeeper** | Pre-commit review — canon, Northstar, quality, security | 0.1 | Gateway on `WAVE_COMPLETE` | Single commit diff |
| **SERAPH** | Security scanning, red-team analysis | 0.4 | Gatekeeper (S1 finding) or Governor | Security surface |
| **QUANTUM** | Research, prior-art, forensic analysis | 0.4 | Governor or Worker (on unknown) | Investigation scope |
| **LÆX** | Canon check, Northstar ratification | 0.2 | Governor or Gatekeeper | Canon + Northstar |

### 1.2 Invariants per role

**Governor**
- Exactly one Governor per build session.
- Holds the authoritative build context object (§5.2).
- Is the only agent that calls `session_start`, `session_end`, and `merge`.
- Escalates to HITL via webshell when `confidence < 0.95` or a categorical exclusion zone is hit.
- Never implements code directly while Workers are active — orchestration and implementation do not mix in the same turn.

**Worker**
- Claims exactly one task at a time.
- Operates exclusively within its assigned worktree (`~/lightarchitects/worktrees/<codename>/`).
- Never touches the primary worktree (`~/Projects/lightarchitects-sdk/`).
- Sends `FILE_CLAIM` before editing any file not in its task `owned_files`.
- Sends `WAVE_COMPLETE` with commit SHA before marking a task complete.

**Gatekeeper**
- Stateless between reviews — no memory of prior reviews.
- Never modifies files; only reads and reports.
- Produces exactly one `GATE_REVIEW` per trigger.
- Temperature 0.1 is mandatory — same diff must always produce the same verdict.

### 1.3 System prompt header (all agents)

Every agent receives a structured header block before its role-specific prompt. This establishes ground truth at spawn time:

```
[AGENT_CONFIG]
role:             <governor|worker|gatekeeper>
temperature:      <0.1|0.2|0.3|0.4>
agent_id:         <role>-<build_codename_slug>-<seq>
session_id:       <uuid from session_start>
build_codename:   <codename>
program_codename: <program or null>
worktree_path:    ~/lightarchitects/worktrees/<codename>
branch:           feat/<codename>
base_commit_sha:  <sha at spawn time>
phase:            <n>
wave:             <n>
playbook_version: 1.1.0
```

The header is machine-generated by the gateway at spawn time. Agents treat it as immutable ground truth — they never override it.

### 1.4 Agent ID conventions

Each agent uses a consistent identifier for `claimant` fields and all A2A messages:

```
<role>-<build_codename_slug>-<seq>

Examples:
  governor-squad-comms-operator
  worker-squad-comms-operator-1
  worker-squad-comms-operator-2
  gatekeeper-squad-comms-operator-1
```

`build_codename_slug` = first 3 words of the codename joined with hyphens.
For `squad-comms-operator-ui`: slug is `squad-comms-operator`.

## Part II — The Build Scope Hierarchy

Every build operation lives at exactly one scope. Scope determines isolation, directory, gate intensity, and what "advance" means. See §15.9 for the full treatment with concrete workflow examples. Summary:

| Scope | Isolation | Path | Gate on advance |
|---|---|---|---|
| **Program** | Long-lived worktree | `~/lightarchitects/worktrees/<program>/` | None (sub-builds gate independently) |
| **Build** | Fresh worktree per lifecycle | `~/lightarchitects/worktrees/<codename>/` | Full Q/S/I/N/D/V |
| **Phase** | Logical boundary in build worktree | Same as build | Q1–Q4 + D2 |
| **Wave** | Commit boundary in build worktree | Same as build | Q1–Q4 (Gatekeeper) |
| **Task** | File-set boundary within wave | Same as build | Q1 lint |

**The primary worktree** (`~/Projects/lightarchitects-sdk/`) stays on `main` at all times. It is the merge operator, not a build environment. Agents never receive this path as `--cwd`.

## Part III — A2A Language Standard

### 3.1 Format: JSONL

All agent-to-agent messages are **JSONL** — one self-contained JSON object per line.

```
Why JSONL:
- Streamable line-by-line (no buffering needed)
- Appendable to log files without rewrites
- Diffable in git
- Unambiguous to parse at low temperature
- Natural fit for the gateway's NDJSON streaming mode
```

### 3.2 Canonical message envelope

Every message, regardless of type or direction, carries this envelope:

```jsonl
{"v":1,"agent_id":"worker-abc123","session_id":"sess-001","build_codename":"squad-comms-operator-ui","program_codename":"weaving-grafting-canon","worktree_path":"~/lightarchitects/worktrees/squad-comms-operator-ui","branch":"feat/squad-comms-operator-ui","commit_sha":"ce01143","phase":1,"wave":1,"task_id":"task-001","timestamp":"2026-05-12T10:30:00Z","type":"<MessageType>","payload":{}}
```

**Required fields:**

| Field | Source | Purpose |
|---|---|---|
| `v` | Constant `1` | Protocol version |
| `agent_id` | System prompt header | Sender identity |
| `session_id` | `session_start()` return | Scopes message to one build run |
| `build_codename` | Task `build_codename` field | Routes to correct build |
| `worktree_path` | `--cwd` spawn argument | Full filesystem path (absolute) |
| `branch` | `git branch --show-current` | Explicit branch identity |
| `commit_sha` | `git rev-parse HEAD` | Point-in-time anchor |
| `type` | Enum (§3.4) | Drives receiver behaviour |
| `timestamp` | ISO-8601 UTC | Ordering and audit |

**Request/response fields** (add when relevant):

| Field | Present in | Purpose |
|---|---|---|
| `request_id` | Requests | UUID for correlation |
| `response_to` | Responses | Echoes `request_id` |
| `confidence` | All non-trivial responses | 0.0–1.0; see §3.3 |
| `citations` | Architecture/security/gate decisions | `[{source, excerpt}]` |
| `verdict` | Gate and review responses | `"accept" \| "reject" \| "hitl"` |
| `error` | Error responses | `{code, message, recoverable}` |

### 3.3 Confidence and citations (Canon XXXV)

Every decision-gating assertion must include:

| Decision type | Required confidence | Citations required |
|---|---|---|
| Wave auto-accept | ≥ 0.95 | N1/N2 + any failing dimension |
| Merge auto-accept | ≥ 0.99 | Full gate dimensions |
| Security finding | ≥ 0.99 | Source + verbatim excerpt |
| Canon check | ≥ 0.95 | Canon section + verbatim excerpt |
| Northstar alignment | ≥ 0.95 | Pillar text + verbatim excerpt |

If confidence falls below the required threshold: `verdict` must be `"hitl"` and `uncertainty_reason` must be included. **Auto-decisions below threshold are forbidden.**

```jsonl
{"type":"GATE_REVIEW","confidence":0.97,"citations":[{"source":"canon://builders-cookbook §7.11","excerpt":"One branch per build. The branch name is feat/{codename} exactly."},{"source":"canon://northstar §P1","excerpt":"operator ships production engineering work E2E from webshell without terminal fallback"}],"verdict":"accept","payload":{}}
```

### 3.4 Message type enum

| Type | Direction | Purpose |
|---|---|---|
| `CONTEXT` | Worker → all | First message after spawn; establishes full scope |
| `QUERY_STATE` | Worker → Governor | Request build state, next task, file owners, canon check |
| `STATE_RESPONSE` | Governor → Worker | Response to any `QUERY_STATE` |
| `FILE_CLAIM` | Worker → all | Declare file ownership for current wave |
| `FILE_RELEASE` | Worker → all | Release ownership after commit |
| `WAVE_START` | Worker → Governor | Beginning wave N |
| `WAVE_COMPLETE` | Worker → Governor | Wave N committed; triggers Gatekeeper |
| `GATE_REVIEW` | Gatekeeper → Governor | Pre-commit review verdict |
| `PHASE_GATE_RESULT` | Governor → all | Phase gate passed/failed |
| `BLOCKER` | Worker → Governor | Cannot proceed; typed blocker |
| `HEARTBEAT` | Worker → Governor | Alive signal; current state snapshot |
| `TASK_HANDOFF` | Governor → Worker | Task assignment with full context |
| `MERGE_READY` | Governor → all | All gates passed; about to merge |
| `MERGE_COMPLETE` | Governor → all | Merged to main; new main commit SHA |
| `SHUTDOWN` | Governor → Worker | Terminate after current task |

## Part IV — The Two State Machines

### 4.1 Build state machine (Governor-owned)

```
session_start()
      │
 [PREFLIGHT]  ← G1–G8 preflight (git-lifecycle-canon §III)
      │  worktree created + branch cut
 [ACTIVE]
      │  tasks enqueued
 [PHASE_N_ACTIVE]
      │
 [WAVE_N_ACTIVE]
      │
   ┌──┴──────────────────┐
wave gate pass        wave gate fail
   │                      │
[WAVE_N_COMPLETE]    [WAVE_N_BLOCKED]
   │                      │  resolve
more waves? ──yes──► [WAVE_N+1_ACTIVE]
   │ no
[PHASE_N_GATE]
   │
   ┌──┴──────────────────┐
phase gate pass      phase gate fail
   │                      │
[PHASE_N_COMPLETE]   [PHASE_N_BLOCKED]
   │
more phases? ──yes──► [PHASE_N+1_ACTIVE]
   │ no
[MERGE_GATE]  ← full Q/S/I/N/D/V
   │
   ┌──┴──────────────────┐
merge gate pass      merge gate fail
   │                      │
[MERGING]            [MERGE_BLOCKED]
   │
[MERGED]
   │
session_end()
   │
[CLOSED]
```

**Session rules:**
- `session_start` is called exactly once per build. The Governor stores `session_id` and passes it to every `add_task` call via `build_session_id`.
- `session_end` is called by the Governor after all tasks reach a terminal state (`completed`, `failed`, `skipped`).
- If the Governor restarts, call `session_start` again with the same `build_codename`. A new UUID is minted — propagate the new `session_id` to all subsequently-enqueued tasks.
- Workers do **not** call `session_start` or `session_end`. They read `build_session_id` from their claimed task.

### 4.2 Task state machine (Worker-reported)

```
add_task()
    │
[PENDING]
    │  claim_task(id, source=<agent-id>)
[IN_PROGRESS]
    │
  ┌─┴───────────────────────────────┐
gates pass                   blocker detected
    │                               │
[COMPLETED]     [AWAITING_OPERATOR_RESOLUTION]
                                    │  resolve_assertion_gate()
                               [IN_PROGRESS]
                                    │
                           [COMPLETED / FAILED]
```

**Task fields relevant to coordination:**

| Field | Type | Set by |
|---|---|---|
| `id` | `string` | Queue (auto-minted on `add_task`) |
| `build_codename` | `Option<string>` | Governor via `add_task` |
| `build_session_id` | `Option<string>` | Governor via `add_task` |
| `assignee` | `Option<string>` | Governor (pre-assign) or Worker (on claim) |
| `status` | `TaskStatus` | Queue transitions |
| `awaiting_assertion_id` | `Option<string>` | Set on blocker raise |
| `resolution_deadline` | `Option<datetime>` | Governor or conductor |

**Priority semantics:**

| Label | Meaning |
|---|---|
| `high` | Security fixes, blockers, gates |
| `medium` | Normal feature work |
| `low` | Cleanup, docs, quality improvements |

Workers always claim the highest-priority `pending` task for their `build_codename` unless the Governor has pre-assigned them a specific task (`assignee` field matches their agent ID).

### 4.3 State persistence

| Layer | Store | Written by |
|---|---|---|
| Runtime (fast) | `~/.lightarchitects/squad_comms/sessions/<id>.json` | Gateway on every transition |
| Runtime (fast) | `~/.lightarchitects/squad_comms/tasks/<id>.json` | Gateway on claim/complete |
| Canonical (durable) | `~/lightarchitects/soul/helix/corso/builds/active.yaml` | `/BUILD` orchestrator on phase transitions only |

Direct edits to `active.yaml` are **forbidden** for agents. Gateway meta-skills own that surface.

### 5.4 Governor orchestration duties

#### Build-start checklist

```
1. session_start(build_codename)               → record session_id
2. Verify preflight gates (disk space, no collision, git-lifecycle-canon §III)
3. add_task() × N for all Wave 1 tasks        → pass session_id on each via build_session_id
4. chat_inject("[Governor] Build <codename> started. N tasks enqueued.")
5. Spawn Worker agents (via agent_stream or external runner)
```

#### Phase transition gates

Between phases, the Governor:
1. Calls `list_tasks` and waits for all current-phase tasks to reach terminal state. Local timeout: 2× estimated phase duration; ≥4h for LARGE tier. The gateway does not enforce phase timeouts in v1 — this is Governor-side responsibility.
2. Runs quality gate: `cargo test`, `cargo clippy`, `pnpm test:run` as applicable. If gate fails, Governor raises a BLOCKER rather than proceeding.
3. Enqueues next-phase tasks via `add_task`.
4. Broadcasts `PHASE_GATE_RESULT` + announces via `chat_inject`.

#### Merge gate

Before merging a branch to main, the Governor verifies:
- All tasks for the build are `completed` (none `pending`, `in_progress`, or `failed`)
- SERAPH scan returns no CRITICAL or HIGH findings
- LÆX canon check passes
- All HITL gates have been resolved
- Full Q/S/I/N/D/V gate passes (git-lifecycle-canon §IV)

See §9.2 for the concrete merge command sequence.

## Part VI — File Ownership Protocol

### 6.1 Ownership chain

```
Governor sets owned_files in add_task metadata (at enqueue — before any Worker starts)
    ↓
Worker inherits at claim_task time
    ↓
Worker sends FILE_CLAIM before editing any file outside that set
    ↓
Worker sends FILE_RELEASE with commit_sha after committing
```

### 6.2 Conflict detection rules

| Scenario | Action |
|---|---|
| Different `build_codename` | No conflict — separate worktrees, filesystem-isolated |
| Same build, different wave | Likely sequencing error → Worker raises `BLOCKER` |
| Same build, same wave, different agent | Governor must reassign one agent's task |

### 6.3 File granularity

| Situation | Scope |
|---|---|
| Single file edit | Claim that file only |
| Module refactor | Claim all files in `src/<module>/` |
| Cross-crate change | Claim per crate; use separate tasks if possible |
| Generated files (`Cargo.lock`, lockfiles) | Never claim — Governor controls via sequential wave ordering |

## Part VII — Pre-commit Gatekeeper

### 7.1 Trigger and spawn

```
Worker sends WAVE_COMPLETE(commit_sha=X)
    → Gateway receives event
    → Gateway spawns Gatekeeper:
        --cwd ~/lightarchitects/worktrees/<codename>
        --commit-sha X
        --context <build_context_object>
        --temperature 0.1
```

The Gatekeeper is spawned into the **same worktree** as the Worker so it can
run `cargo clippy`, `cargo test`, and `git diff HEAD~1` directly.

### 7.2 Eight review dimensions

| Dim | Gate | Auto-fail condition |
|---|---|---|
| **Q1** | `cargo fmt --check` | Any formatting diff |
| **Q2** | `cargo clippy -- -D warnings` | Any warning-as-error |
| **Q3** | `cargo test` | Any test failure |
| **Q4** | lizard cyclomatic ≤ 10, fn ≤ 60 lines | Any violation |
| **N1** | Northstar pillar advanced | No pillar advancement in diff |
| **N2** | Canon compliance | `unwrap`, `panic!`, nested `feat/` branch, etc. |
| **S1** | Security — no secrets, no injection, no timing side-channels | Any finding severity ≥ HIGH |
| **D1** | All new public items have doc comments | Any missing doc on `pub` item |

### 7.3 Auto-accept / reject / HITL decision table

| Condition | Decision |
|---|---|
| All 8 dimensions pass + confidence ≥ 0.95 | **AUTO-ACCEPT** |
| Any Q1–Q4 fail | **AUTO-REJECT** + fix-it task created |
| Any S1 finding severity ≥ HIGH | **HITL** regardless of other results |
| N1 or N2 fail + confidence ≥ 0.95 | **AUTO-REJECT** + canon guidance posted |
| N1 or N2 fail + confidence < 0.95 | **HITL** |
| D1 fail | **AUTO-REJECT** (no HITL — add the doc comment and re-commit) |
| confidence < 0.95 on any dimension | **HITL** |

**Why 95% for waves, 99% for merge?**
Wave commits are intermediate state — the full Q/S/I/N/D/V merge gate is the
safety boundary for `main`. Over-gating waves kills velocity. Under-gating the
merge gate risks production.

### 7.4 `GATE_REVIEW` message

```jsonl
{"v":1,"agent_id":"gatekeeper-001","session_id":"sess-001","build_codename":"squad-comms-operator-ui","worktree_path":"~/lightarchitects/worktrees/squad-comms-operator-ui","branch":"feat/squad-comms-operator-ui","commit_sha":"1a2b3c4","phase":1,"wave":1,"timestamp":"2026-05-12T10:35:00Z","type":"GATE_REVIEW","confidence":0.97,"verdict":"accept","payload":{"dimensions":{"Q1_fmt":"pass","Q2_clippy":"pass","Q3_tests":{"result":"pass","tests_run":386,"tests_fail":0},"Q4_complexity":"pass","N1_northstar":{"result":"pass","pillar_advanced":"pillar_1_e2e_from_ui"},"N2_canon":"pass","S1_security":{"result":"pass","findings":[]},"D1_docs":"pass"},"summary":"All 8 dimensions pass.","auto_decision":"accept"}}
```

### 7.5 HITL checkpoint flow

```
GATE_REVIEW(verdict=hitl)
    → Gateway pauses build (state: HITL_PENDING)
    → Webshell shows operator checkpoint:
        ┌─────────────────────────────────────┐
        │  Diff viewer                        │
        │  Gate dimension results             │
        │  Gatekeeper uncertainty_reason      │
        │  Canon citations                    │
        │  Northstar pillar impact            │
        ├─────────────────────────────────────┤
        │ [Accept] [Reject] [Modify] [Override with justification] │
        └─────────────────────────────────────┘
    → Override is always logged to audit/gate-overrides-<date>.log
      {operator_id, justification, commit_sha, dimension, confidence}
```

### 7.5.1 Frontend HITL Surface Implementation Contract (Ratified 2026-05-30 per Canon XXXIX, LÆX #56)

Every webshell HITL surface that displays an agent escalation **must ship with a resolution
action path**. Display-only rendering — showing escalation data with no Approve/Reject action —
is not a valid HITL implementation. It provides false assurance to operators while the
underlying agent remains blocked indefinitely.

**Three requirements (all mandatory):**

1. **Action path required** — every HITL card renders at minimum an Approve and Reject action.
   The resolution call must be a POST to a named backend endpoint, not a fire-and-forget event.

2. **Typed subscription for new events** — legacy event pipelines that feed display-only arrays
   (e.g., `la:escalation` window event → `pendingEscalations` in Cockpit.svelte) must not be
   repurposed for new agent action flows. New agent HITL events use a dedicated typed SSE
   subscription via `subscribeByTopic()`.

3. **Nonce-validated resolution** — resolution endpoints for agent actions must validate a
   server-minted escalation nonce (UUIDv7) to prevent replay attacks. The nonce is embedded in
   the resolution request body and verified server-side against a replay-kill set. The nonce must
   never appear in logs or error messages (Security Guardrails §3.3, CWE-209).

**Violation class**: a HITL surface that renders agent escalation data but carries no resolution
action path is a [S] gate violation, routed to SERAPH.

### 7.6 Reserved Gate-Label Namespace + Amendment-Label Convention (Ratified 2026-05-13 per Canon XXXIX)

The following labels are CANONICAL and RESERVED — they must NOT be used for any other purpose (e.g., SCRUM amendment IDs, ad-hoc review-finding tags, plan-internal tracking IDs). Re-purposing reserved labels causes vocabulary collision that obscures gate semantics in plans, manifests, and audit logs.

**Reserved canonical labels** (do NOT re-use):

| Namespace | Owner | Reserved tokens |
|-----------|-------|-----------------|
| Pre-flight gates | Agents Playbook §15.3 | `G1`, `G2`, `G3`, `G4`, `G5`, `G6`, `G7`, `G8` |
| Pre-commit Gatekeeper dimensions | Agents Playbook §7.2 | `Q1`, `Q2`, `Q3`, `Q4`, `N1`, `N2`, `S1`, `D1` |
| Pre-merge gate categories | Agents Playbook §15.4 | `Q`, `S`, `I`, `N`, `D`, `V` (followed by digit) |
| Pre-finalization rubric anchors | Architects Blueprint Part XIV | `C1`, `C2`, `C3`, `C4`, `C5`, `C6`, `C7`, `C8` |
| Gate vocabulary (Canon XXXVIII) | Platform Canon | `[A]`, `[S]`, `[Q]`, `[C]`, `[O]`, `[P]`, `[K]`, `[D]`, `[T]`, `[R]` |

**Canonical amendment-label namespaces** (USE these instead):

| Use case | Namespace | Pattern |
|----------|-----------|---------|
| SCRUM Round N amendments | `SCRn-N` | `SCR1-1`, `SCR1-2`, ..., `SCR1-10`, `SCR2-1`, ... |
| Canon Audit Round N amendments | `CARn-N` | `CAR3-1` (R3 first amendment), `CAR3-2`, ... |
| Operator-flagged amendments (out-of-band) | `OFn-N` | `OF1-1`, `OF1-2`, ... |
| Future review tier amendments | `<TIER>n-N` prefix | e.g., `REVn-N` for /REVIEW amendments |

**Rationale**: Demonstrated 2026-05-13 in the `gateway-action-audit-claude-runtime` plan — a plan used "G1-G10" labels for SCRUM Round 1 amendments. The collision with Playbook §15.3 pre-flight gates "G1-G8" was caught only via Canon Audit Round 3 by the Playbook auditor agent, requiring an after-the-fact `labeling_clarification` frontmatter block to disambiguate. This canonical reservation pre-empts the collision at amendment-label-creation time.

**Operational rule**:
- Any new amendment-label scheme MUST declare its namespace in the plan/manifest frontmatter (`labeling_convention` field) IF it does not use one of the canonical namespaces above.
- Plan authors MUST NOT re-use reserved tokens.
- Review-tier authors (SCRUM, Canon Audit, /REVIEW) MUST use their canonical prefix.

**Extension (2026-05-19 — artifact-internal Gn collision, candidate E2 from vault-migration-v1 soak-close audit)**:

Within a **single artifact** (manifest.yaml, gate-evaluation report, soak_window block), the prefix `G<N>` MUST resolve to exactly ONE namespace among `{check_id, phase_gate, scrum_amendment}`. If two namespaces co-exist in the same artifact, the secondary namespace MUST use a disambiguator prefix:
- `G1-check`, `G2-check` … for numbered check IDs (e.g. g1-report.yaml `checks` block)
- `G1-phase`, `G2-phase` … for phase-gate labels (e.g. soak_window `gate:` field)
- `G1-scrum`, `G2-scrum` … for SCRUM-round amendments

**Demonstrated by**:
- 2026-05-13 incident: `gateway-action-audit-claude-runtime` plan G1-G10 SCRUM labels collided with Playbook §15.3 pre-flight G1-G8 (caught Canon Audit Round 3).
- 2026-05-19 incident: `vault-migration-v1/g1-report.yaml` used G1-G14 as `checks` IDs AND G2/G8 as `soak_window.gate` labels in the same artifact — discovered during LÆX soak-close audit. The g1-report's "G2 + G8 must hold continuously" line was unparseable without convention-disambiguator.

**Rationale**: Cross-artifact collision is recoverable via `labeling_clarification` frontmatter; artifact-internal collision is not — both namespaces share the same parse surface. Disambiguator prefix is the only fix that doesn't require future-context.

**Composition**: Canon XXXVIII reserves the gate-vocabulary tokens `[A+S+Q+C+O+P+K+D+T+R]` semantically. §7.6 reserves the operational labels (`G1-G8`, `Q1-Q4`, `N1-N2`, `S1`, `D1`, `C1-C8`) syntactically. Together they protect the canonical vocabulary at both the semantic and naming layers.

### 7.7 Post-Implementation Hypothesis Verification

**Principle**: Code produced by a spawned Worker is a *hypothesis*. The Worker's local gates (Q1-Q4: fmt/clippy/test/complexity) verify compilation and surface regressions, but they cannot evaluate correctness of the implementation against the task spec. An independent semantic code review by a `lightarchitects:<reviewer_agent>` is mandatory after every Worker task completion and before WAVE_COMPLETE.

**Trigger**: Governor receives the Worker's completion signal (`IMPLEMENTATION_COMPLETE`) for a task.

```
Worker completes task → IMPLEMENTATION_COMPLETE signal
    → Governor reads reviewer_agent from wave declaration in plan
    → Governor spawns: Agent(subagent_type="lightarchitects:<reviewer_agent>",
                              prompt="Review the diff at commit <sha> for task <task_id>.
                                      Task spec: <task.prompt>.
                                      Assess: correctness, architecture fit, error handling,
                                              doc completeness, security surface.
                                              File scope: <task.owned_files>.
                                              Report REVIEW_PASS or REVIEW_FAIL with findings.")
    → Review Agent returns verdict
```

**Verdict routing**:

| Verdict | Condition | Action |
|---|---|---|
| `REVIEW_PASS` | confidence ≥ 0.95, no blocking findings | FILE_RELEASE; task status → `completed`; count toward WAVE_COMPLETE eligibility |
| `REVIEW_FAIL` | Any blocking finding OR confidence < 0.95 | Create fix-it task for same Worker; do NOT FILE_RELEASE; do NOT count toward WAVE_COMPLETE |
| `review_hitl` | confidence < 0.95 on verdict itself | AskUserQuestion with diff + findings before routing |

**Reviewer domain defaults** (from Blueprint §23.8):

| Wave content | `reviewer_agent` |
|---|---|
| General implementation | `lightarchitects:quality` |
| Security-touching (auth/crypto/permissions) | `lightarchitects:security` (mandatory) |
| New public APIs / exported types | `lightarchitects:quality` |
| Observability / tracing / metrics | `lightarchitects:ops` |
| Test suites | `lightarchitects:testing` |

If `reviewer_agent` is not declared in the plan wave, Governor defaults to `lightarchitects:quality`.

**REVIEW_FAIL fix-it cycle**:
1. Governor creates a fix-it task with the REVIEW_FAIL findings as the task prompt.
2. The fix-it task targets the same Worker (preserves FILE_CLAIM ownership — prevents conflicting edits from other agents).
3. Worker implements fixes, re-commits, signals IMPLEMENTATION_COMPLETE again.
4. Governor spawns Review Agent on the new diff.
5. Maximum 2 consecutive REVIEW_FAIL cycles on the same task before escalating to HITL. Repeated failure indicates spec ambiguity or capability ceiling, not just an implementation error.

**What review covers** (beyond Gatekeeper mechanical Q1-D1 dimensions):
- Task spec compliance: does the implementation match what was asked?
- Logic correctness: inversions, off-by-one, unhandled cases, race conditions
- Architecture fit: consistent with surrounding conventions; no unnecessary abstractions
- Error handling completeness: all error variants mapped, no silent swallows
- Security surface: no unintended new trust boundaries in the diff
- Doc completeness: all public items have meaningful (not just present) doc comments

**Relationship to Gatekeeper (§7.2)**: The Gatekeeper runs mechanical checks *after* WAVE_COMPLETE. Hypothesis verification runs *before* WAVE_COMPLETE. They are sequential, not alternatives. A wave's sequence is: IMPLEMENTATION_COMPLETE → review → (on pass) FILE_RELEASE → WAVE_COMPLETE → Gatekeeper.

**Enforcement**: `/BUILD` Step 7.2 (SQUAD dispatch) is responsible for the IMPLEMENTATION_COMPLETE → review → WAVE_COMPLETE routing. Workers MUST NOT send WAVE_COMPLETE directly — only the Governor emits WAVE_COMPLETE after all tasks in the wave have received REVIEW_PASS.

## Part VIII — Async Coordination Patterns

### 8.1 Worker startup sequence

```
spawn
  1. list_tasks()
     → filter client-side by build_codename=<my_build>
     → find pending tasks with my assignee, or highest-priority pending
     → record build_session_id from the target task record
  2. claim_task(id=<id>, source=<agent-id>)
     → verify status transitions to "in_progress"
     → on 409 CONFLICT: list_tasks again, claim next available
  3. chat_inject(session_id=<build_session_id>,
                 message="[<agent-id>] Claimed task <id>: <title>. Starting.",
                 sender=<agent-id>)
  4. send CONTEXT message
  5. QUERY_STATE(next_task) → receive STATE_RESPONSE
  6. FILE_CLAIM(files, wave)
  7. implement
```

If no tasks are available: `chat_inject` to announce readiness, then poll `list_tasks` every 30 seconds until work appears or Governor sends `SHUTDOWN`.

### 8.2 Status update cadence

Workers inject status messages into the soul-chat session at these milestones:

| Milestone | Message format |
|---|---|
| Claim | `[<agent-id>] Claimed task <task_id>: <title>. Starting.` |
| Phase boundary | `[<agent-id>] Phase <N> complete. <N> tests passing. Next: <description>.` |
| Gate result | `[<agent-id>] Gate: <gate_name> — PASS / FAIL (<detail>).` |
| Blocker | `[<agent-id>] BLOCKER on task <task_id>: <description>. Needs: <what>.` |
| Completion | `[<agent-id>] Task <task_id> complete. Commit: <sha>. <files_changed> files.` |

**Rate limit:** no more than one status message per 60 seconds, unless the message is a BLOCKER or completion announcement.

> **v1 caveat:** `chat_inject` writes to the webshell's in-memory session store. Cross-process delivery is non-functional until the session store migrates to a durable backend. Treat `chat_inject` as best-effort audit logging in v1; do not block on acknowledgement from the SSE stream.

### 8.3 Parallel Workers (same build, different waves)

```
Worker A (Wave 1)                Worker B (Wave 2)
    │                                 │
FILE_CLAIM(wave=1,files=[X,Y])        │
    │                      FILE_CLAIM(wave=2,files=[P,Q])
    │  ← each receives the other's claim; no overlap → proceed
implement                         implement
    │                                 │
WAVE_COMPLETE(wave=1,sha=abc)         │
    │                      WAVE_COMPLETE(wave=2,sha=def)
```

Governor verifies no `owned_files` overlap at `add_task` time. File overlap
across parallel Workers is a **Governor enqueue error**, not a Worker error.

### 8.4 Phase gate — the one synchronisation point

```
Governor waits for ALL tasks in phase → [COMPLETED or FAILED]
    │
runs Q1–Q4 against worktree
    │
broadcasts PHASE_GATE_RESULT
    │
pass → enqueue next phase tasks → Workers resume async
fail → Workers receive SHUTDOWN or targeted fix-it task
```

### 8.5 Blocker semantics

A blocker is any condition that prevents a Worker from making forward progress without external input. The Worker must not spin-wait — it raises the blocker and parks.

**Blocker raise sequence:**
```
1. Worker detects blocker condition.
2. Worker sends BLOCKER message (type="BLOCKER", payload={description, needs}).
3. Worker calls:
   chat_inject(session_id, "[<agent-id>] BLOCKER on task <task_id>: <desc>. Needs: <what>.")
4. Worker parks: terminates cleanly. Does NOT poll, sleep, or retry.
   Task transitions to awaiting_operator_resolution via conductor assertion gate.
```

**Blocker routing by type:**

| Blocker type | Routes to | Resolution |
|---|---|---|
| HITL gate (operator decision required) | Operator via webshell HITL panel | `resolve_assertion_gate` |
| Technical blocker (dependency on another task) | Governor via `query_blocked_flow` | Governor enqueues dependency task |
| Security finding blocking merge | Governor → SERAPH → LÆX | Governor gates merge; resolves or escalates |
| Ambiguous spec | Governor → Operator | Governor injects clarifying question |

**Blocker resolution flow:**
```
GOVERNOR                              OPERATOR
  │
  ├── query_blocked_flow(build_id=<build_id>)
  │   ← list of tasks in awaiting_operator_resolution
  │
  ├── Evaluate blocker type (table above)
  │
  │   [if HITL gate]
  ├── chat_inject("Operator: <question>. Please resolve via /assert <assertion_id>.")
  │                                         │
  │                                         └── operator calls
  │                                             resolve_assertion_gate(
  │                                               request_id, assertion_id, build_id,
  │                                               operator_id, action_type, citation?)
  │
  │   [task status transitions back to pending]
  │
  ├── (new Worker claims and continues)
```

If `resolution_deadline` expires before resolution: task transitions to `failed`. Governor detects on next `list_tasks` poll and escalates to operator via `chat_inject`.

### 8.6 Crashed Worker recovery

1. No `HEARTBEAT` for > 5 min → Governor calls `task_logs(id)`.
2. Governor reads `git log` in worktree — if commits exist past `last_known_commit_sha`, work is partially complete.
3. Governor creates replacement task with `metadata.resume_from_commit = <last_sha>`.
4. New Worker does HEAD sha check → fast-resumes from that commit.

**Idempotency and fault tolerance:**

| Failure | Recovery |
|---|---|
| Governor restarts | `session_start` is idempotent; Governor calls `list_tasks` to reconstruct state; incomplete tasks remain claimable |
| Worker restarts | Worker calls `list_tasks`, finds its own task `in_progress`. If same agent ID in `source`, continues. A 409 on re-claim means another Worker took the task — find next pending. |
| Webshell down | MCP tools return `GatewayError::InvalidRequest`; agents back off 30s and retry; 3 consecutive failures → Governor raises HITL gate |
| Soul-chat inject fails | Non-fatal: log the failure but do not block implementation work. Status can be inferred from task state. |

> **v1 limitation:** No programmatic task release. Tasks stuck `in_progress` by a crashed Worker require operator intervention (manual queue edit or webshell admin panel).

## Part IX — Git Decision Protocol

### 9.1 Pre-commit checklist (Worker)

Before every `git commit`:
1. `git rev-parse HEAD` → capture current `commit_sha`
2. `git status --porcelain` → verify only `owned_files` are staged
3. `cargo fmt --check` → Q1 must pass
4. **Canon review (BLOCKING gate)** — before WAVE_COMPLETE, read and verify implementation
   against all 7 canon documents (`canon://platform-canon`, `canon://builders-cookbook`,
   `canon://agents-playbook`, `canon://architects-blueprint`, `canon://operators-manual`,
   `canon://lasdlc-template`, `canon://security-guardrails`). Report any violations as
   `CANON_VIOLATION: [gate] <description>` in the WAVE_COMPLETE payload.
   Severity: BLOCKING (halts merge) | WARNING (surfaced for operator decision).
5. Send `WAVE_COMPLETE` with full gate results (including canon_violations field)

### 9.2 Pre-merge checklist (Governor)

Before `git merge --no-ff`:
1. `git -C ~/Projects/lightarchitects-sdk branch --show-current` == `main`
2. All Workers have sent `FILE_RELEASE` for all claimed files
3. All tasks are `[COMPLETED]` — no `[IN_PROGRESS]` or `[PENDING]`
4. Full Q/S/I/N/D/V gate passes (§15.4)
5. Broadcast `MERGE_READY` — wait one poll cycle for final blockers
6. Execute:
```bash
git -C ~/Projects/lightarchitects-sdk merge --no-ff feat/<codename>
git -C ~/Projects/lightarchitects-sdk push github main
git -C ~/Projects/lightarchitects-sdk worktree remove ~/lightarchitects/worktrees/<codename>
git -C ~/Projects/lightarchitects-sdk branch -d feat/<codename>
```
7. Broadcast `MERGE_COMPLETE` with `main_sha_before`, `main_sha_after`, `worktree_removed: true`.

### 9.3 File path convention in all messages

```
✅  "lightarchitects-gateway/src/agent_stream/mod.rs"    ← worktree-relative
✅  "~/lightarchitects/worktrees/squad-comms/"            ← worktree root (absolute OK)
❌  "/Users/kft/lightarchitects/worktrees/.../mod.rs"     ← absolute with $HOME (breaks on other operators)
```

## Part XV — Git Topology & Lifecycle

> *"Order is the shape upon which beauty depends."* — Pearl S. Buck

This part codifies the complete lifecycle of feat/branches and worktrees — from pre-flight gates that authorize creation, through quality gates that authorize merge, to cleanup protocols that retire them safely. It standardizes how to recognize a **clean topology**, how to **return to one**, and which **guardrails** protect against the destructive incidents (soul-engine, sdk-divergence) that prompted this canon.

*Originated from the 2026-05-11 cleanup session (18 stale branches, 5 stashes, 1 locked worktree) and the squad-comms-session-per-build workflow walkthrough. Absorbed from git-lifecycle-canon v1.1.*

---

### 15.1 Clean topology spec

#### Worktree topology

A **clean topology** at any point in time satisfies ALL of:

1. **Exactly one canonical main worktree** at a deterministic path. For lightarchitects-sdk, this is `~/Projects/lightarchitects-sdk/`.
2. **Long-lived program worktrees** only for programs with multi-week lifespans, placed under `~/lightarchitects/worktrees/<program>/`.
3. **Build-scoped worktrees** at `~/lightarchitects/worktrees/<codename>/`, one per active feat/branch, created on kickoff and removed on merge.
4. **Agent-isolated worktrees** at `~/lightarchitects/worktrees/agent-<hash>/`, locked while active, removed by the spawning agent on completion.
5. **Primary worktree** stays on `main` at all times. Never checked out to a feature branch except transiently during a merge operation — switches back to `main` immediately after push. Workers and build agents receive only a linked worktree path; they never touch the primary worktree.
6. **No orphan worktrees** — every entry in `git worktree list` maps to either an active build (in active.yaml), a paused build (with documented resumption criteria), or the canonical main.

#### Branch topology

A clean branch topology satisfies ALL of:

1. **No stale branches** — branches whose content is fully on main (0 unique commits) are deleted within one operator session of merging.
2. **No stale tracking refs** — `git fetch --prune` is run as part of any cleanup pass.
3. **All active branches map to active.yaml** — every `feat/<codename>` corresponds to a build entry, or is the operator placeholder.
4. **Naming convention enforced** (see §15.2).
5. **No commits on the primary worktree's branch except merge commits**.

#### Stash topology

1. **Stashes are shared across worktrees** — they live at `.git` level, not per-worktree.
2. **Every stash has a descriptive message** — `git stash push -m "<descriptive-name-with-date>"`.
3. **Stashes are time-bounded** — anything older than 30 days needs a decision.
4. **Real work is committed, not stashed** — stashes are for transient interruptions only.

### 15.3 Pre-flight gate (`/preflight-worktree`)

Required BEFORE allocating any new worktree + branch.

| Gate | Check | Fail action |
|---|---|---|
| **G0 — primary clean** | `git status --short \| wc -l` equals 0 on primary worktree | Abort: WIP on primary violates §15.3 invariant. Run §15.5.7 Triage subprotocol first to preserve work, then retry G1–G8. |
| **G1 — main reachability** | `git fetch github main` succeeds | Abort: investigate network/auth |
| **G2 — main sync** | local main `==` github/main | Abort: integrate remote first |
| **G3 — build clean** | `cargo check --workspace --all-features` clean | Abort: main is broken — fix before forking |
| **G4 — quality clean** | `make quality` passes (fmt + clippy + tests) | Abort: main has quality regressions |
| **G5 — name collision** | branch + path do not already exist | Abort: rename or remove the colliding artifact |
| **G6 — active.yaml mapping** | active.yaml has an entry for `<codename>` (or operator confirms placeholder) | Warn: create entry before commits accumulate |
| **G7 — parent program** | if `<parent-program>` specified, exists in active.yaml | Abort: typo or missing parent |
| **G8 — disk space** | ≥10GB available for new worktree | Abort: free disk first |


#### Primary worktree isolation invariant (non-negotiable)

> **The primary worktree is the merge operator, not a build environment.**

|---|---|---|

### 15.4 Pre-merge gate (`/gate`)

Required BEFORE merging worktree → main. Five gate families:

#### Q — Quality (auto, CI-enforced)
- **Q1** `cargo fmt --check` clean
- **Q2** `cargo clippy --all-targets --all-features -- -D warnings` clean
- **Q3** `cargo test --workspace --all-features` passes
- **Q4** Builders Cookbook compliance: no `unwrap()`/`panic!()` in production, complexity ≤10, function ≤60 lines

#### S — Security (auto-routed via specialized agents)
- **S1** SERAPH scan on the branch diff
- **S2** `sonatype` + `cargo-deny` if Cargo.toml changed
- **S3** LÆX Layer 1 + Layer 4 review if any security-touching code (no operator override)
- **S4** No `--no-verify`, no `--no-gpg-sign`, no force-push to main

#### I — Integration (auto)
- **I1** Branch rebased on current main (no stale base)
- **I2** No merge conflicts
- **I3** Post-rebase build still passes

#### N — Northstar fit (auto + LÆX)
- **N1** `northstar_lineage` block present in build plan
- **N2** `pillar_advanced` claim validated by review predicate
- **N3** No categorical exclusion zone unaddressed

#### D — Documentation (auto + manual)
- **D1** Public APIs documented (`missing_docs = "deny"`)
- **D2** Build plan updated to reflect completion state
- **D3** active.yaml updated via `/BUILD` orchestrator (NOT direct edit)

#### V — Verification (manual + auto)
- **V1** E2E tests pass for affected surfaces
- **V2** Smoke test post-deploy (if deployment changes)

#### Merge protocol

On FULL PASS:
```bash

git -C ~/Projects/lightarchitects-sdk merge --no-ff <branch>
git -C ~/Projects/lightarchitects-sdk push github main
git -C ~/Projects/lightarchitects-sdk worktree remove ~/lightarchitects/worktrees/<codename>
git -C ~/Projects/lightarchitects-sdk branch -d <branch>
```

**Pre-condition**: verify `git -C ~/Projects/lightarchitects-sdk branch --show-current` == `main` before merging. If it is not, stop and restore before proceeding.

**Never** `--no-verify`, **never** `--no-gpg-sign`, **never** force-push to main.

On any FAIL:
- Governor posts fix-it notes via squad-comms `chat_inject`
- Worker addresses, re-runs `/gate`
- Loop until full PASS

#### 15.4.5 Post-Commit Tree Verification (mandatory after agent-orchestrated commits)

Trust the tree, not the exit code. After every `git commit` executed by an agent,
verify content actually landed:

```bash
PARENT_TREE=$(git rev-parse HEAD~1^{tree})
HEAD_TREE=$(git rev-parse HEAD^{tree})
if [ "$PARENT_TREE" = "$HEAD_TREE" ]; then
    echo "❌ PHANTOM EMPTY COMMIT detected — HEAD tree identical to parent."
    echo "   Likely cause: pre-commit hook auto-formatted/re-staged into matching HEAD."
    echo "   Action: investigate before continuing — bisects will hit a no-op."
fi
```

**Why mandatory**: Pre-commit hooks that auto-format (e.g., `cargo fmt` re-staging)
or auto-discard transient changes can produce **content-identical commits** —
commit object exists, tree matches parent, no `--allow-empty` was specified.
The commit object exists but contributes nothing. Reflog hits a no-op. Reviewers
waste cycles. This is invisible to `git commit` exit code, and invisible to
`git log --oneline`.

**Symmetric-diff trap**: `git diff A..B --stat` and `git diff B..A --stat` are
**inverse views of the same delta**, not evidence of bi-directional content.
To assess true containment, use:

```bash
git ls-tree -r A | sort > /tmp/a.tree
git ls-tree -r B | sort > /tmp/b.tree
comm -23 /tmp/a.tree /tmp/b.tree   # tree entries in A not in B
```

Empty output = B is a superset (or equal) of A.

**Evidence**: CORSO 2026-05-13 — commit `524cc38` succeeded with `git commit`
exit 0, full pre-commit hook output `=== Quality gates passed ===`, but
`git ls-tree -r` revealed tree-identity with parent `f44997d`. Recovered via
`git reset --hard` after EVA cross-examination caught the symmetric-diff
misinterpretation.

### 15.5 Cleanup protocol (with guardrails)

#### 15.5.1 Layer 1 protection (mandatory)

Per `memory://feedback_destructive_git_prevention`: **commit working code OR stash with descriptive name BEFORE any cleanup operation**.

```bash
git stash push --include-untracked -m "<purpose>-pre-cleanup-<date>"
```

#### 15.5.2 Per-action approval (canonical)

Per `memory://feedback_no_destructive_git`: these operations require **explicit, scoped, in-the-current-turn approval — even in auto-mode**:

- `git clean` (any form)
- `git reset --hard`
- `git checkout .` / `git checkout -- <path>`
- `git stash drop` / `git stash clear`
- `git branch -D`
- `git push --delete` to remotes
- `git push --force` / `--force-with-lease` to any branch
- `git worktree remove --force`

**Batch approval** is acceptable IFF the operator explicitly approves the batch in the current turn. EVA delegation does NOT extend to destructive git ops.

#### 15.5.3 Verification before classification

Before classifying a branch as "safe to delete":

1. **File-content verification trumps commit-log inference** — grep main directly for unique content.
2. **`git fetch --prune` first** — local tracking refs lie.
3. **Check uncommitted worktree state** — worktree may contain real work not on branch HEAD.

#### 15.5.4 Standard cleanup sequence

```
1. Inventory   (read-only): git worktree list; git branch -a; git stash list; git status
2. Classify    (read-only): SAFE / INFERRED-SAFE / HAS-UNIQUE-WORK / PAUSED / ACTIVE
3. Protect     (mandatory): stash any worktree with uncommitted state
4. Approve     (HITL gate): present categorized findings; operator approves per-action or per-batch
5. Execute     (after approval): use audit-logged script pattern
6. Verify      (read-only): re-run inventory; confirm clean topology spec (§15.1) holds
```

#### 15.5.5 Audit logging

Every destructive cleanup pass logs to `~/.lightarchitects/audit/git-cleanup-<date>.log`:
- Pre/post inventory (worktrees, branches, stashes)
- Each operation with target + timestamp
- Recovery hashes for stash drops and branch deletes

#### 15.5.6 Recovery

All destructive git ops are **reversible within ~90 days** via reflog:

```bash
# Recover a dropped stash:
git fsck --unreachable | grep commit && git stash apply <hash>

# Recover a deleted branch:
git reflog && git checkout -b <recovered-branch> <hash>

# Uncommitted worktree state: NOT RECOVERABLE if no stash was made.
```

#### 15.5.7 Triage subprotocol — accumulated WIP archive + feature-slice separation

`/UNTANGLE` orchestrates this; can also be run by hand. Use when:

- ≥10 modified files, OR ≥5 untracked files, OR mixed staged+unstaged+untracked state
- WIP sits on primary (`main`) — violates §15.3 invariant
- WIP straddles multiple unrelated features and needs per-feature review

This subprotocol **preserves** work (unlike §15.5.4 which **removes** orphans).
Cleanup deletes; Triage partitions. They are different operations with different
mental models.

##### Phase 1 — Pre-flight Inventory (mandatory, read-only)

Capture the safe-return state BEFORE any branch operations. Persist to
`/tmp/triage-<repo>-<date>.log` — recovery anchor.

```bash
git log --oneline -5                                  # current HEAD chain
git rev-parse origin/<base>                           # last pushed point
git stash list                                        # existing stashes
git reflog -10                                        # recent moves
git status --short | wc -l                            # WIP scope
git diff --stat | tail -2                             # unstaged delta
git diff --cached --stat | tail -2                    # staged delta
git ls-tree -r HEAD | wc -l                           # current tree size
```

##### Phase 2 — Archive Snapshot (mandatory, foreground only)

Create a holding archive branch with the FULL WIP atomically:

```bash
git checkout -b triage/<repo>-wip-archive-<YYYYMMDD>
git add -A
git commit --no-verify -m "chore(archive): WIP snapshot <date> — pre-triage

Do NOT merge. Holding archive per §15.5.7."
```

**Critical rules**:
- Use `triage/` prefix — signals DO NOT MERGE (not a `feat/` candidate)
- `--no-verify` permitted (per `feedback_no_verify_two_tier`: Tier A — archives)
- **Foreground only**: NEVER `run_in_background: true` for any branch-state op.
  Background `commit && checkout` produces race conditions where the checkout
  races ahead, leaving the staging area inconsistent. CORSO 2026-05-13
  phantom-empty-commit incident traced to this.
- Verify via §15.4.5 post-commit tree check before returning to base
- Chain operations with `&&` in a single Bash call when possible

##### Phase 3 — Return to Clean Base

```bash
git checkout <base>                                   # main or origin/<base>
git status --short | wc -l                            # MUST be 0
git ls-tree -r HEAD | wc -l                           # confirm tree matches base
```

If status is non-zero, untracked files survived. Investigate before continuing.

##### Phase 4 — Feature-Slice Mapping (HITL gate)

Before creating feature branches, present the proposed grouping for approval:

```
Repo: <repo>
Files in archive: <N total>
Proposed feature groups:
  feat/<repo>-license       — N files (license attribution stack)
  feat/<repo>-<feature-A>   — N files (...)
  feat/<repo>-<feature-B>   — N files (...)
  ...

Continue? [yes/revise]
```

HITL gate prevents incorrect groupings from being committed and reviewed under
the wrong feature lens.

##### Phase 5 — Feature-Slice Branches (per-feature)

For each approved group:

```bash
git checkout -b feat/<repo>-<feature-name>            # from base, not archive
git checkout triage/<repo>-wip-archive-<date> -- <files for this feature>
git commit -m "feat(<feature>): <description>"        # hook runs normally
# Verify via §15.4.5 tree check
git checkout <base>
```

**Rules**:
- Cut branches from `<base>` (main), NOT from the archive or a parent feature branch
- One feature per branch — never combine unrelated features
- `--no-verify` policy per `feedback_no_verify_two_tier`:
  - Tier A (archives): blanket OK
  - Tier B (feature branch + environmental failure): OK with documentation
  - Code-defect failures: NEVER silently bypassed — flag BLOCKING in review task

##### Phase 6 — Containment Verification

Use ls-tree, NOT symmetric diff stats (per §15.4.5):

```bash
git ls-tree -r triage/<repo>-wip-archive-<date> | sort > /tmp/archive.tree
for branch in $(git branch --list 'feat/<repo>-*' | grep -v archive); do
  git ls-tree -r "$branch" | sort
done | sort -u > /tmp/features.tree
comm -23 /tmp/archive.tree /tmp/features.tree         # entries in archive missing from features
```

Empty output = full containment. Any output = files missing from a feature
branch (likely an oversight in grouping).

##### Phase 7 — Review Task Registration

Create one `TaskCreate` per feature branch with the canonical review template:

- **Northstar alignment**: which Pillar (P1: webshell E2E / P2: vibe-coding orchestration / both / neither)
- **Canon compliance**: which canon docs the branch must satisfy
- **Active plan cross-check**: does an existing build plan cover this work?
- **Fullstack viability**: does the feature stand alone or require dependent work?
- **Decision criteria**: keep / rework / drop

##### Forbidden during triage

- `git stash --include-untracked` on heavy WIP — loses the staged/unstaged distinction
- `git reset --hard` on the source branch before archive verification (§15.4.5)
- `run_in_background: true` on any branch-state operation
- `git checkout -- <file>` to discard unstaged work before archive exists
- Combining branch-state operations across multiple Bash invocations when
  staging-area state persists between calls (use `&&` chains instead)

##### Recovery from triage failure

The archive branch is the recovery anchor. If any feature-slice step goes wrong:

```bash
git checkout <base>
git reset --hard <base@{1}>                           # reflog-based; per-action approval
git branch -D feat/<repo>-<problem-feature>           # per-action approval
# Archive at triage/<repo>-wip-archive-<date> is untouched — start over.
```

**Evidence**: 2026-05-13 multi-repo triage session — protocol successfully
applied across 5 repos (lightarchitects-cli, CORSO-DEV, EVA, AYIN-DEV,
lightarchitects-next), producing 32 reviewable feature branches with zero
WIP loss after recovering from the CORSO phantom-empty-commit incident.

### 15.7 Verification methodology

Canonical hierarchy of evidence for branch cleanup classification:

1. **Highest** — direct file-content existence on main (`grep` / `ls` against main worktree)
2. **Medium** — `git log main..<branch>` empty AND `git diff main...<branch>` empty
3. **Low** — "0 commits ahead" with no diff verification
4. **Unreliable** — commit-message inference, PR-number references

For **mixed-divergence** branches (simultaneously older AND newer than main):

```bash
for f in <suspect-files>; do
  INS=$(git diff main -- "$f" | grep -c "^+[^+]")
  DEL=$(git diff main -- "$f" | grep -c "^-[^-]")
  NET=$((INS - DEL))
  echo "$f → +$INS -$DEL  net=$NET"
done
```

- **Net + values** → potentially UNIQUE work not on main
- **Net - values** → working tree is OLDER than main
- **~Zero net + high ins/del** → formatting reorganization (rustfmt artifacts)

### 15.8 Key learnings (canonized 2026-05-11)

1. **Local tracking refs lie.** Always `git fetch --prune` first. The "phantom remote" state caused 14 of 18 `push --delete` calls to fail before pruning revealed reality.

2. **Stashes are shared across worktrees.** They live at `.git` level, not per-worktree. The "15 stashes" misclassification (actually 5 unique) added 200% to the inventory plan.

3. **Worktree-cwd lock.** A process whose cwd is inside a worktree blocks `git worktree remove`. Cleanup must start from outside the worktree being removed.

4. **"Verified safe to delete" still needs HITL.** Canon overrides confidence. Even at 99.99% verification confidence, destructive ops remain at their canonical authority level.

5. **Layer 1 protection is non-negotiable.** Stash with descriptive name BEFORE any destructive op. The soul-engine incident — 3+ hours of reconstruction work — happened because `git clean -fd` destroyed untracked files.

6. **Mixed-divergence detection.** A worktree's working tree can be simultaneously older AND newer than main — per-file net delta analysis catches this. Pure "diff vs branch HEAD" misses it.

7. **File-content verification > commit-log inference.** Always grep main directly to verify "this content shipped". Commit messages and PR references rot.

8. **Audit log every destructive operation.** `~/.lightarchitects/audit/git-cleanup-<date>.log` with pre/post state and recovery hashes. Cheap insurance.

9. **Per-batch approval > per-action when verification is uniform.** When 18 branches all have the same canonical justification, one batch approval suffices. Per-action creates approval fatigue without adding safety.

10. **Cleanup is a build, not an aside.** It deserves a plan, a Wave, a /gate, and an audit log. Half-doing it creates worse state than not doing it.

11. **`git add -u` before `git rebase --continue`.** After resolving rebase conflict hunks, `git ls-files -u` returning empty does NOT guarantee a clean stage. Auto-merged files (e.g., `Cargo.lock`) may be modified but unstaged. Always run `git add -u` to stage ALL modified tracked files before `git rebase --continue`, or the rebase refuses with "must resolve conflicts" even when no conflict markers remain. Evidence: EEF E2 shell-and-output rebase, 2026-05-13.

### 15.9 Build scope hierarchy — the universal loop

> *The same loop runs at every scope. What changes is isolation depth, gate intensity, and what "advance" means.*

#### The five scopes

```
Program          (weeks–months)
  └── Build      (days–week)      ← LASDLC lifecycle
        └── Phase (hours–day)     ← logical grouping of waves
              └── Wave  (30–90min) ← atomic commit boundary
                    └── Task  (<30min) ← individual work item
```

**Scope is not optional hierarchy** — it maps directly to:
1. **Isolation mechanism** (new worktree vs existing worktree vs commit)
2. **Directory path** (which `~/lightarchitects/worktrees/` entry)
3. **Gate intensity** (full Q/S/I/N/D/V vs subset)
4. **Advance action** (merge to main vs phase boundary commit vs wave commit vs task commit)

#### Scope → isolation → directory mapping

| Scope | Isolation | Path | Branch |
|---|---|---|---|
| **Program** | Long-lived linked worktree | `~/lightarchitects/worktrees/<program>/` | `feat/<program>` (long-lived) |
| **Build** | Fresh linked worktree per lifecycle | `~/lightarchitects/worktrees/<codename>/` | `feat/<codename>` (merge → delete) |
| **Phase** | Logical boundary within build worktree | same as build | same as build |
| **Wave** | Commit boundary within build worktree | same as build | same as build |
| **Task** | File-set boundary within wave | same as build | same as build |

> **Rule**: New worktree = new branch. Phases, waves, and tasks do NOT get their own branches — they are commit milestones on the build's branch. This keeps the git graph linear and the merge operation simple.

#### The universal loop (fractal)

```
PREPARE  →  WRITE  →  GATE  →  ADVANCE
```

| Scope | PREPARE | WRITE | GATE | ADVANCE |
|---|---|---|---|---|
| **Program** | G1–G8; create program worktree | Sub-builds proceed independently | N/A | Sub-builds merge to main |
| **Build** | G1–G8; create build worktree + branch | Agent works in `~/la/worktrees/<codename>/` | Full Q/S/I/N/D/V | `merge --no-ff` → main; worktree removed |
| **Phase** | Read existing worktree state; identify wave set | Waves execute within the phase | Q1–Q4 + D2 | Phase-boundary commit |
| **Wave** | Agent reads files scoped to the wave | Implements deliverables; commits atomically | Q1–Q4 (Gatekeeper) | `git commit -m "feat: <wave-label>"` |
| **Task** | Agent checks file ownership | Implements task; may be multiple sub-commits | Q1 (fmt) | Sub-commit or squash into wave commit |

#### Concrete workflow examples

**Program: EEF**
```bash
# One-time program setup
git -C ~/Projects/lightarchitects-sdk worktree add \
    -b feat/embodied-engineering-forge \
    ~/lightarchitects/worktrees/embodied-engineering-forge \
    github/main
# Program worktree survives all sub-build merges.
# Sub-builds each get ~/lightarchitects/worktrees/<sub-build-codename>/ branching from github/main.
```

**Build: `squad-comms-session-per-build`**
```bash
git -C ~/Projects/lightarchitects-sdk branch --show-current  # → must be main
git -C ~/Projects/lightarchitects-sdk worktree add \
    -b feat/squad-comms-session-per-build \
    ~/lightarchitects/worktrees/squad-comms-session-per-build \
    github/main
lightarchitects agent start \
    --cwd ~/lightarchitects/worktrees/squad-comms-session-per-build \
    --codename squad-comms-session-per-build
# ... agent runs phases + waves ...
git -C ~/Projects/lightarchitects-sdk merge --no-ff feat/squad-comms-session-per-build
git -C ~/Projects/lightarchitects-sdk push github main
git -C ~/Projects/lightarchitects-sdk worktree remove \
    ~/lightarchitects/worktrees/squad-comms-session-per-build
git -C ~/Projects/lightarchitects-sdk branch -d feat/squad-comms-session-per-build
```

**Sub-build Ship within Program (e.g., EEF E2 `shell-and-output` → `feat/embodied-engineering-forge`)**
```bash
# 1. In the sub-build worktree: rebase onto current parent branch tip
git -C ~/lightarchitects/worktrees/shell-and-output fetch github
git -C ~/lightarchitects/worktrees/shell-and-output rebase github/feat/embodied-engineering-forge
#    Per conflict round: resolve hunks → git add -u → git rebase --continue

# 2. Push sub-build branch (force-with-lease is authorized — rebase rewrites history)
git -C ~/lightarchitects/worktrees/shell-and-output \
    push github feat/eef/shell-and-output --force-with-lease

# 3. Discard any files blocking primary checkout (e.g., .remember/tmp/save-session.pid, Cargo.lock)
git -C ~/Projects/lightarchitects-sdk checkout -- .remember/tmp/save-session.pid Cargo.lock

# 4. Switch primary worktree to the parent program branch (transient merge-operator step)
git -C ~/Projects/lightarchitects-sdk checkout feat/embodied-engineering-forge

# 5. Merge sub-build --no-ff into parent (not into main)
git -C ~/Projects/lightarchitects-sdk merge --no-ff feat/eef/shell-and-output

# 6. Push parent branch
git -C ~/Projects/lightarchitects-sdk push github feat/embodied-engineering-forge

# 7. Restore primary worktree to main immediately
git -C ~/Projects/lightarchitects-sdk checkout main

# 8. Update plan frontmatter: phase → next, shipped phase status → COMPLETE
```

**Key invariants for sub-build Ship**:
- Sub-build merges into the **parent program branch**, not into `main`. `main` receives the program when the full program ships.
- `--force-with-lease` on the sub-build push is the correct scope: authorized for own branch, forbidden for parent or main.
- Primary worktree returns to `main` immediately after step 6 — the transient checkout (steps 4–6) is merge-operator-only.
- Shared extension points (server.rs dispatch, mod.rs declarations, stores.ts, events/types.rs) always conflict when sibling sub-builds extend the same registries. Take the incoming sub-build content for new tool registrations; keep HEAD for variants/stores that postdate the conflict.

#### Simplification: one parameterized command

```bash
lightarchitects build start <codename> --scope [program|build|phase|wave|task]
```

| `--scope` | Creates worktree? | Creates branch? | Gate on advance |
|---|---|---|---|
| `program` | Yes (long-lived) | Yes (long-lived) | None |
| `build` | Yes (lifecycle) | Yes (merge → delete) | Full Q/S/I/N/D/V |
| `phase` | No | No | Q1–Q4 + D2 |
| `wave` | No | No | Q1–Q4 |
| `task` | No | No | Q1 |

**Why preserve the hierarchy rather than flatten it?**
1. **Directory isolation is load-bearing.** Builds must NOT share a worktree (file-lock collisions, rollback scope).
2. **Gate intensity is correct, not arbitrary.** A task-level Q1 lint check is not a build-level S1 SERAPH scan.
3. **Rollback scope matches scope.** Wave fails → revert wave commit. Build fails → abandon branch.
4. **Multi-agent safety.** Multiple agents within the same wave coordinate only if scope is explicit.

## Part XVI — Parallel Dispatch Principles

> *"The tool-level DAG parallelizer and the agent-level SQUAD parallelizer solve the same problem — which independent units of work can execute concurrently? — at different scales."*

**Core principle**: Partition by file ownership, not by feature. Features cross file boundaries. Two agents editing the same file from different feature perspectives create merge conflicts.

### The Five Principles

**1. File-Ownership Partitioning**

Before spawning agents, map every task to the files it will touch. Group tasks so:

```
For all agents A_i, A_j where i != j:
  files(A_i) ∩ files(A_j) = ∅
```

If two tasks must touch the same file, they go to the same agent — or the shared file is split into non-overlapping sections with explicit coordination.

**2. Dependency-Ordered Batching**

```
Level 0 (Foundation):  Types, configs, utilities — no deps on other agents
Level 1 (Consumers):   Code that uses Level 0 types — depends on Level 0
Level 2 (Integration): Code that wires Level 0+1 together — depends on both
```

Within a level, all agents run in parallel. Between levels, sequential.

**3. Explicit Exclusion Rules**

Every agent prompt must state what it may NOT touch:

```
Agent 1: "Add #[instrument] to llm/mcp modules.
  EXPLICITLY EXCLUDE src/tui/ — 40fps = 200+ span events/sec."

Agent 3: "Create NEW files only (ayin/, tui/ayin_waterfall.rs).
  Do NOT modify existing files in src/llm/ or src/agent/ — Agent 1 owns those."
```

**4. Shared-File Section Contracts**

When two agents must touch the same file, define explicit section boundaries:

```
runner.rs ownership:
  Agent 1: Lines with #[instrument] attributes + debug!() macro additions
  Agent 2: AgentEvent::Complete break paths + gate_permissions()
  Contract: Agent 1 adds attributes ABOVE function signatures.
            Agent 2 modifies control flow INSIDE function bodies.
```

State the contract in both agents' prompts.

**5. The DAG Isomorphism**

The same algorithm applies at every scale:

| Scale | Units | Dependencies | Algorithm |
|---|---|---|---|
| Tool calls | Individual tool invocations | Same-file writes | `DependencyGraph::infer()` + Kahn's BFS |
| Agent dispatch | SQUAD agent prompts | Shared file ownership | Manual partition + level batching |
| Build phases | Plan phases | Type/API dependencies | Sequential with parallel sub-phases |

### Pre-Dispatch Checklist (mandatory before spawning)

1. **Map files**: for each task, list files created or modified.
2. **Compute partitions**: group tasks so no two agents share a file.
3. **Order by dependency**: assign agents to levels; Level 0 runs first.
4. **State exclusions**: each agent prompt includes an explicit "DO NOT TOUCH" list.
5. **Verify partition**: confirm `files(A_i) ∩ files(A_j) = ∅` for same-level agents.

**Cargo fmt rule**: agents in the same level MUST NOT run `cargo fmt` on files outside their ownership set. Only the orchestrator runs `cargo fmt` after all agents in a level complete.

**Evidence** (lÆx0-cli Phase 6, 2026-04-04): 3 parallel agents, 0 merge conflicts, 294 tests, ~10 min wall-clock vs ~30 min sequential.

## Part XVII — Squad Synthesizer Protocol

> *"By the mouth of two or three witnesses every word shall be established."* — 2 Corinthians 13:1

The Squad Synthesizer is the **fan-in algorithm** that aggregates 7 independent `gate_evaluation` blocks (one per Gatekeeper agent) into a single `squad_review.yaml` verdict at every LASDLC phase boundary.

Gate dimensions: **9** — [A][S][Q][C][O][P][K][D][T][R]

### Input Contract

Gate evaluation blocks are written to `<build_root>/.gate-evals/<phase-id>-<gate>.yaml`. Required fields per block: `schema_version`, `gate`, `scored_by`, `scored_at`, `build_id`, `phase_id`, `overall_verdict`, `scored_anchors`.

### Synthesis Algorithm (5 Steps)

**Step 1 — Schema validation**: verify all required fields; verify `scored_by` matches `gatekeeper-registry.yaml`; verify `build_id`/`phase_id` consistent; verify GAPS/FAIL verdicts have `verbatim_anchor_quote` (Canon XXXV). Failure → `VALIDATION_ERROR`. Stop.

**Step 2 — Veto application**:

| Authority | Sibling | Trigger | Effect |
|---|---|---|---|
| `lightarchitects:security` | SERAPH | `overall_verdict == FAIL` on `[S]` | `squad_review.verdict = FAIL` |
| `lightarchitects:knowledge` | SOUL | `overall_verdict == FAIL` on `[K]` OR citation violation | `squad_review.verdict = FAIL` |
| `lightarchitects:quality` | LÆX | `overall_verdict == FAIL` on `[C]` | `squad_review.verdict = FAIL` |

Multiple vetoes may apply simultaneously. All are recorded.

**Step 3 — GAPS counting**:

| gaps_count | verdict_tier |
|---|---|
| 0 | `PASS` |
| 1–2 | `GAPS_NOTED` |
| ≥3 | `FIXES_REQUIRED` |
| Any FAIL in scored_anchors | `FAIL` |

**Step 4 — Conflict detection**: when two agents cite the same anchor UUID with different verdicts, record as `disputes[]` for HITL review. Security/Knowledge primary anchors take precedence without escalation.

**Step 5 — LDB aggregation**: merge `ldb_components` across all blocks:
- `low: min(all low values)`, `point: mean(all point values)`, `high: max(all high values)`.

### Output

Written to `<build_root>/.squad/squad-review.yaml`. Verdict semantics:

| verdict | Build continues? | HITL required? |
|---|---|---|
| `PASS` | Yes | No |
| `GAPS_NOTED` | Yes | No (gaps logged, fix before wave-close) |
| `FIXES_REQUIRED` | No | Yes — Kevin approves override |
| `FAIL` | No | Yes — mandatory; veto blocks continuation |

### Minimum Quorum

Security `[S]` (SERAPH) and Knowledge `[K]` (SOUL) evaluations are **always required** for a `PASS` verdict. Absence → `FAIL`. `[R]` (QUANTUM) is non-quorum — absence is recorded with a warning.

### Reference Implementation

```bash
python3 scripts/synthesize-squad-review.py --phase <phase-id> --build-root <path>
```

Exit codes: `0` = PASS/GAPS_NOTED, `1` = FIXES_REQUIRED, `2` = FAIL/VALIDATION_ERROR.

### Why Veto Asymmetry?

Security and Knowledge hold veto power because their failure modes are **non-composable**: a single authentication bypass or fabricated citation cannot be averaged out by ten passes elsewhere.

## Part XVIII — Agent Architecture Standard

> Every agent follows the 3-layer prompt pattern. Skills define methodology. New capability = new skill file, not new agent.

### The 3-Layer Prompt Architecture

```
┌──────────────────────────────────────────────────┐
│ LAYER 1: IDENTITY (static)                        │
│ WHO you are — voice, strands, principles,          │
│ anti-patterns. Never changes between dispatches.  │
├──────────────────────────────────────────────────┤
│ LAYER 2: DOMAIN (static)                          │
│ WHAT you work on — category of work, supported    │
│ modes, domain rules, skill references.             │
├──────────────────────────────────────────────────┤
│ LAYER 3: MISSION (dynamic — injected at dispatch) │
│ WHAT to do NOW — mode, target, skill, prior        │
│ context, constraints. NOT in agent file.           │
└──────────────────────────────────────────────────┘
```

**Layer 1 — Identity** (static, in agent .md file): sibling name + voice rules, core strands, output principles, anti-patterns. Identical regardless of task.

**Layer 2 — Domain** (static): class of inputs, supported modes, domain-specific rules, skill references. Narrows expertise without specifying a task.

**Layer 3 — Mission** (dynamic, injected at dispatch): MODE (methodology), SKILL (which skill file), TARGET (path or target), CONTEXT (prior findings), CONSTRAINTS (restrictions). The agent file contains only a placeholder section.

### Agent Taxonomy

| Type | When to use | Rules |
|---|---|---|
| **Singleton Template** (default) | Single task, fresh per dispatch | Stateless; new capability = new skill file, not new agent |
| **Orchestrator** (exception) | Decomposes complex engagements, dispatches templates | At most ONE per sibling; must have `Agent` tool |
| **Daemon** (exception) | Continuous monitoring, threshold-based escalation | Rare — only for real-time monitoring |
| **Governed Executor** (exception) | Authorization gate per action, legal/compliance constraints | Scope/auth governance baked in |

### Agent Rules

1. Every sibling has AT MOST one orchestrator agent.
2. Singleton templates use 3-layer prompts. All three layers explicitly labeled.
3. Skills define methodology, not agents. Agent = hands. Skill = brain. Mission = briefing.
4. **New capability = new skill file, not new agent.**
5. Diverge from Singleton Template ONLY for Orchestrators, Daemons, or Governed Executors — each exception justified.
6. Every sibling has exactly 1 user-invocable meta-skill. All sub-skills are `user-invocable: false`. Exception: SCRUM is dual-purpose.

### Decision Tree — Adding New Capability

1. Can an existing template handle this with a new MODE? → Add a mode, not an agent.
2. Can a new SKILL provide the methodology? → Add a skill, not an agent.
3. Does this need a genuinely different DOMAIN (Layer 2)? → New template justified.
4. Does this need to orchestrate other agents? → Orchestrator justified (limit 1 per sibling).
5. Does this need continuous autonomous operation? → Daemon justified (rare).

If 1–2 answer is yes, DO NOT create a new agent.

### §8.7 Recursion Termination Invariant (Helix Walkers)

Every recursive helix walker MUST satisfy two independent termination conditions:

**Condition 1 — Bounded depth**:

```rust
const MAX_FS_HELIX_DEPTH: u8 = 7;     // filesystem helix nesting
const MAX_TRAVERSAL_DEPTH: u8 = 7;    // graph traversal (Neo4j path)
```

These are separate constants. Walkers MUST NOT reuse the other's constant — different domains.

**Condition 2 — Visited-set with canonical identity**:

Track visited entries by **canonicalized inode tuple** `(dev: u64, ino: u64)`, NOT helix-id or path string. Path strings are spoofable via symlink chains; inode tuples are the only identity that survives symlink loops.

```rust
let mut visited: HashSet<(u64, u64)> = HashSet::new();
fn walk(path: &Path, depth: u8, visited: &mut HashSet<(u64, u64)>) -> Result<()> {
    if depth >= MAX_FS_HELIX_DEPTH {
        return Err(HelixError::DepthExceeded);
    }
    let canonical = path.canonicalize()?;
    let meta = canonical.metadata()?;
    let key = (meta.dev(), meta.ino());
    if !visited.insert(key) {
        return Err(HelixError::CycleDetected { at: canonical });
    }
    // ... recurse into children with depth + 1
}
```

**Required tests** — every walker implementation MUST include:
- `test_depth_termination()` — synthetic deep tree at depth 8+ returns `DepthExceeded`
- `test_cycle_detection()` — synthetic symlink loop returns `CycleDetected`, not infinite recursion

**Applies to**: filesystem walkers (helix.toml discovery), graph traversers (Cypher path patterns), 3D viz drill-down (setScope animation — the camera's scope-history stack must use inode tuples to prevent loops if the underlying graph has a cycle).

```
1. Call session_start(build_codename="<BUILD_CODENAME>") → record the returned session_id.

- All tasks for the build are `completed` (zero pending, in_progress, or failed).
- SERAPH scan returns no CRITICAL or HIGH findings.
- LÆX canon check passes.
- All HITL gates have been resolved.

If any condition fails: raise a HITL gate. Do NOT merge.

- Monitor the soul-chat SSE stream for BLOCKER messages.
- On BLOCKER: call list_tasks() and filter for build_codename="<BUILD_CODENAME>" with status in
  ["awaiting_operator_resolution", "failed"] to identify blocked tasks. (query_blocked_flow is
  webshell-HTTP-only in v1 and cannot be called as an MCP tool — use this list_tasks filter instead.)
  - Technical dependency blocker → enqueue the dependency task; re-queue the blocked task.
  - HITL gate (operator decision required) → escalate (see Escalation below).
  - Security finding → halt the affected task; route to SERAPH scan; do not merge.
  - Ambiguous spec → inject clarifying question into chat; await operator response.
- On Worker silence (no status update for >2× the task's estimated duration) → query task_logs and assess. If stuck: raise HITL.

## Escalation (HITL) Format

When operator input is required:
  chat_inject(session_id, "@operator [HITL] <one-sentence description>. Options: a) <option> b) <option> c) <option>. Awaiting your choice.", sender="governor-<SLUG>")

Wait for operator response via chat before proceeding. Log the decision in the build's chat thread.

## Status Cadence

- Maximum 1 chat_inject per 60 seconds for routine status.
- HITL escalations and completion announcements bypass the rate limit.
- Hourly: inject a brief build status summary (tasks done/in_progress/pending, Workers active, any HITL queue).

- Webshell unreachable: back off 30 seconds and retry. After 3 consecutive failures → raise HITL to operator.
- Governor restart: call session_start(same build_codename) — a new session_id will be minted (NOT
  idempotent: each call returns a different UUID). Propagate the new session_id to all subsequent
  add_task and chat_inject calls. Call list_tasks to reconstruct current build state. WARNING: tasks
  added in the prior session retain their old build_session_id. Before each chat_inject, re-read
  build_session_id from the relevant task record via list_tasks rather than using a cached value.
- chat_inject failure: non-fatal. Log locally. Do not block implementation work on chat delivery.

## Authority Hard Limits — NEVER do these

- Do NOT delete worktrees you did not create.
- Do NOT run `git push --force`, `git reset --hard`, or `git clean -f` without operator approval.
- Do NOT override an operator veto on a HITL gate.
- Do NOT spawn more Workers than the operator approved.
- Do NOT merge a branch if any merge gate condition is unmet.
- Do NOT make security or compliance trade-off decisions unilaterally — always escalate.

## Session Close

When all tasks are in a terminal state and the merge gate passes:
1. Call session_end(session_id).
2. Inject: "[Governor] Build <BUILD_CODENAME> complete. All tasks terminal. Merge gate passed. Session closed."
```

```
### Prepare

- Identify integration points and naming conventions — match them exactly.
- For multi-file tasks, grep for existing patterns before introducing new ones.

- **Git lifecycle:** follow canon://agents-playbook#part-xv for all branch, commit, and worktree operations.

### Gate

- **Rust:** `cargo test && cargo clippy -- -D warnings && cargo fmt --check`
- **Svelte/TypeScript:** `pnpm test:run && pnpm exec svelte-check --threshold error`
- If a gate fails: fix the root cause and re-gate. NEVER mark a task complete with failing gates.

### Repeat
If any gate fails or code doesn't integrate: fix root cause, re-gate, loop until fully correct.

| Milestone | Format |
|-----------|--------|

| Blocker | `[worker-<id>] BLOCKER on task <task_id>: <description>. Needs: <what>.` |

3. **Wait for the review result.** Do NOT claim the next task yet.
   - **REVIEW_PASS**: inject `[worker-<id>] Task <task_id> review PASS. Claiming next.` → call list_tasks() and claim the next pending task (return to Startup step 1).
   - **REVIEW_FAIL**: receive the fix-it task from the Governor → implement the listed fixes → re-commit → return to step 1 of Completion. Maximum 2 fix-it cycles before escalating to HITL.
4. If no tasks remain pending after REVIEW_PASS: inject readiness and enter Idle Waiting.

> **Why wait for review?** Your implementation is a hypothesis. The Governor routes an independent `lightarchitects:<reviewer_agent>` to verify correctness, architecture fit, and error handling. You never emit WAVE_COMPLETE directly.

## Blocker Protocol

A blocker is any condition that prevents forward progress without external input.

When you hit a blocker:
1. Call chat_inject with the BLOCKER format: `[worker-<id>] BLOCKER on task <task_id>: <description>. Needs: <what>.`
   **v1 fallback:** because chat_inject is not delivered cross-process in v1, also call task_logs to
   append the BLOCKER description to the task's log output so the Governor can discover it via polling.
2. Do NOT poll, spin-wait, or retry.
3. Terminate cleanly and leave the task state for the Governor to resolve.

Blocker types:
- Ambiguous spec / missing context → raise blocker, describe what's needed.
- Another task must complete first → raise blocker, name the dependency.
- Security concern blocking merge → raise blocker, describe the finding.
- Operator input required → raise blocker with `[HITL]` prefix.

## Idle Waiting

When no tasks are available:
1. Inject: `[worker-<id>] No pending tasks. Ready and waiting.`
2. Call list_tasks() every 30 seconds until a task appears or the Governor sends a termination signal via chat.
3. **Idle timeout:** if no task has appeared after 30 minutes, inject
   `[worker-<id>] Idle timeout reached (30 min). No tasks claimed. Shutting down.` and end the session.
4. On Governor termination signal: inject `[worker-<id>] Acknowledged termination. Shutting down.` and end the session.

## Concerns (non-blocking issues)

If you notice something wrong that doesn't block your task — a security smell, a spec inconsistency, a quality issue — inject a `[concern]` message rather than a BLOCKER:
`[worker-<id>] [concern] task <task_id>: <description>.`

The Governor will decide whether to act on it.

## Authority Hard Limits — NEVER do these

- Do NOT make policy decisions (sequencing, scope, security trade-offs) — escalate via [concern] or BLOCKER.
- Do NOT spawn other Workers. The Governor allocates all Workers.
- Do NOT modify active.yaml, portfolio.md, or any build-tracking artifact directly.
- Do NOT delete worktrees you did not create.
- Do NOT run `git push --force`, `git reset --hard`, or `git clean -f`.
```

## §HITL-7 — Escalation Notification Invariant (2026-05-18 ADDITION)

**Invariant**: every L4 (User) escalation in the autonomous-mode decision pipeline MUST emit an operator-visible event. Silent escalation = Pillar 4 violation (CWE-223 Omission of Security-relevant Information).

**Required components**:

1. **AYIN span**: `escalation.notify { build_id, reason, severity, requires_ack, dispatched_at, ack_deadline_ms }` — child of `escalation.raise` parent span
2. **Schema-typed**: structured payload (never free-form text); fields enforce required vs optional
3. **Non-suppressible**: operator preference can throttle frequency (notification-fatigue config), but never globally disable
4. **Multi-channel**: webshell SSE toast (red banner with REVIEW click-option) + `osascript -e 'display notification'` for off-screen ops + optional Telegram/SMS webhook for off-device alerting
5. **MTTA-measurable**: `escalation.notify.ack_received_at` timestamp populated when operator acknowledges; AYIN computes mean-time-to-acknowledge per build

Without this invariant, autonomous-mode runs that L4-escalate at 3am have no operator-visible signal — operator misses escalation, build hangs indefinitely. CWE-223 + Pillar 6 (operator-legible arc) double-violation.

**Composition with §7.7**: §7.7 governs Worker code review BEFORE WAVE_COMPLETE; §HITL-7 governs escalation AFTER review failure (≥3rd FixAgent iteration per Cookbook §64.3).

## §15.3a — Tier Separation Principle (Supervisor / Worker / Git, 2026-05-18 ADDITION)

**Source**: ironclaw-architecture.html §4 Tiers — Supervisor / Worker / Git zero-LLM separation.

In autonomous-mode builds, three tiers operate at structurally separate concurrency surfaces:

| Tier | Roles | Slot model |
|---|---|---|
| **Supervisor** | Operator's Claude Code session + 8 LightArchitect domain agents + Canon Decision Agent | Async, NEVER in worker pool; spawned on-demand for HITL |
| **Worker** | 7-slot pool: ReviewGate + FixAgents + Worker agents (Sonnet/Ollama/Haiku model routing) | All 7 slots; spawned via AgentRunner builder; serialized via MergeAgent mutex for ref-mutating ops |
| **Git** | MergeAgent + Worktree Manager + Orchestrator scheduler | ZERO LLM — pure git2 + `Command::new("git")`; serializes all git ops; one process at a time against `.git/objects` |

**Invariants** (mechanical):

1. Supervisor and LightArchitects are NEVER part of the 7-slot worker pool. Confusing them produces slot starvation under HITL load.
2. Workers SPAWN via fail-closed AgentRunner builder (Cookbook §65); no permission matrix = no spawn.
3. Git tier has ZERO LLM calls. Pure git2 reads (head, branch) + `Command::new("git")` mutating ops via mutex. Deterministic.
4. Each tier has its own failure-recovery domain: supervisor restart resets HMAC subkeys; worker death frees slot; git lock timeout retries with jittered exp (Cookbook §64.2).

**Anti-pattern**: collapsing tiers (e.g., supervisor running git ops in-band) cross-contaminates failure domains. A git lock collision shouldn't pause supervisor decision-making; an LLM rate-limit shouldn't block git cleanup.

## §11.3a — Canon-as-Cached-System-Prompt (2026-05-18 ADDITION)

**Source**: ironclaw-architecture.html §8 Autonomous Delegation + §10 ReviewGate canon-as-system-prompt pattern.

For long-running supervisor sessions and ReviewGate evaluators, canon docs MUST be delivered as **cached system prompt content**, not per-call user messages:

1. Load 7 canon docs (~80K combined tokens, see canon corpus subset list) at session start
2. Send as `system: [{type: "text", text: <canon>, cache_control: {type: "ephemeral"}}]` per Anthropic prompt-caching API
3. Subsequent calls hit cache at ~10% of base input cost (cache_read_input_tokens per usage response)
4. Use 1-hour TTL (2× write cost, 12× fewer refreshes for bursty load) over default 5-min for supervisor sessions

**Why system not user**: cache-hit semantics only fire when the cached prefix is in `system`, not in `messages`. Putting canon in the user message means 100% input cost per call.

**Composition with §SG-CRYPTO.3**: canon corpus integrity verified via SHA256 of canon files at session start; mismatch HALTS before any L1 decision fires.

**Cost arithmetic** (per Task #17 R2 verified):
- 80K canon + 2K fresh per call + 1K output, 95% cache hit, 97 calls over 72-task program: **~$7.31**
- 70% cache hit: **~$13.94**
- No cache: **~$25.32**
- Cache savings: 3.46× vs no-cache; cost ceiling cannot exceed $100 under spec'd parameters

## §15.3.13 — Wave Dispatch Protocol (2026-05-18 ADDITION, candidate #15)

**Scope**: Orchestrator mechanics for wave-level fan-out/fan-in within a phase. Complements §15.3.x single-task git lifecycle (existing) and Cookbook §64 mutex mechanics (existing). This section formalizes how the orchestrator dispatches N parallel tasks per wave, waits for all-merge, and advances to the next wave.

### §15.3.13.1 Wave-cut invariant

**Wave-N branches MUST be cut from `feat/{build}` HEAD AFTER Wave-(N-1) merges to feat — never before.** This is non-negotiable per Cookbook §64.4 and Ironclaw §9.

```
git rev-parse feat/{build}  ←  HEAD AFTER Wave-N-1 merge complete
git checkout -b task/{build}/p{N}-w{M}-{slug} {feat-head}  ←  ONE per task in Wave-M
```

**Anti-pattern**: cutting all wave branches upfront from feat-build-start HEAD creates merge conflicts when Wave-1 lands changes Wave-2 expected. Wave branches must inherit Wave-(N-1) artifacts via merge-then-cut sequencing.

### §15.3.13.2 Fan-out dispatch

Per wave declaration in plan §17.5 (LASDLC v2.5.2 `waves[].tasks[]`):

1. Orchestrator reads `parallelism: N` from wave block (max 7 per Ironclaw §6 slot pool cap)
2. Creates N worktrees under `<build_worktree_root>/p{N}-w{M}-{slug}/`
3. Spawns N AgentRunner instances in parallel via fail-closed builder (Cookbook §65)
4. Each AgentRunner receives:
   - W3C `traceparent` env var (observability-canon §1.1)
   - HKDF subkey-id for current wave (security-guardrails §SG-CRYPTO.2)
   - Task-specific context bundle (Cookbook §66 tier 1/2/3, 15K cap)
   - File-ownership manifest (no overlap with sibling tasks in same wave)
5. Orchestrator monitors via supervisor channel; emits `WorkerSlotGauge` WebEvent per slot occupancy change

### §15.3.13.3 Fan-in convergence

Wave N+1 dispatch BLOCKED until:
1. All N tasks in Wave-N signal `IMPLEMENTATION_COMPLETE` (per §7.7)
2. All N tasks pass per-task post-implementation review (per §7.7)
3. All N task branches merge to `feat/{build}` via MergeAgent (Cookbook §64 serialized mutex)
4. ReviewGate verdict PASS at wave boundary (Cookbook §64.3; MAX_GATE_ITERATIONS=3)
5. `decisions.md` entry written with `wave_complete` verdict + active subkey-id

**Failure recovery**: any task FAIL → FixAgent dispatched (max 3 iterations); 3rd consecutive FAIL → §HITL-7 escalation, wave-N+1 stays blocked, supervisor pauses dispatcher.

### §15.3.13.4 Wave-internal task dependencies

Tasks within a wave MAY declare `depends_on: [task_id]` referencing tasks in **prior waves**. Within-wave dependencies are disallowed — same-wave tasks MUST be parallel-safe (no shared file ownership; no inter-task communication). If a task within a wave needs the output of another task in the same wave, split into two waves.

### §15.3.13.5 Pre-Dispatch Checklist Tables (2026-05-18 ADDITION — 28 hardcoded git-aware gates)

Operationalizes the wave-cut invariant + fan-out + fan-in into **28 explicit gates** fired pre-phase / pre-wave / pre-task / post-task / post-wave. Each gate is BLOCKING — failure HALTS dispatch + emits operator HITL. Mirrors /BUILD skill v2 Steps 11.3.0-11.3.5 for canonicity.

#### Pre-Phase (4 gates, fires once per phase entry)

| Gate | Check | Failure → |
|------|-------|-----------|
| **PP-1** | feat/{build} clean (no uncommitted) | BLOCK + HITL |
| **PP-2** | No stale `task/{build}/p{phase-1}-*` branches | BLOCK + prune |
| **PP-3** | feat HEAD SHA matches prior-gate-exit SHA | BLOCK + HITL (drift) |
| **PP-4** | `<build_worktree_root>` exists + writable + sentinel | BLOCK + create |

#### Pre-Wave (7 gates, fires per wave BEFORE branch cuts)

| Gate | Check | Failure → |
|------|-------|-----------|
| **PW-1** | Wn-1 fully merged + ls-tree containment (§15.4.5) | BLOCK |
| **PW-2** | No orphan `task/{build}/p{N}-w{M-1}-*` branches | BLOCK + prune |
| **PW-3** | Worktree count = 1 main + 0 task | BLOCK + cleanup §64.7 |
| **PW-4** | Record `WAVE_PARENT_SHA = git rev-parse feat/{build}` (atomic) | non-failing |
| **PW-5** | Each task in wave has unique branch slug | BLOCK + reject plan |
| **PW-6** | Wave tasks have disjoint `file_ownership` | BLOCK + reject plan |
| **PW-7** | wave.parallelism ≤ 7 (Ironclaw §6 slot cap) | BLOCK + reject plan |

#### Pre-Task (7 gates, fires per task BEFORE AgentRunner spawn)

| Gate | Check | Failure → |
|------|-------|-----------|
| **PT-1** | Branch name regex `^task/[a-z0-9-]+/p\d+-w\d+-[a-z0-9-]+$` | BLOCK + reject |
| **PT-2** | Task branch cut from `$WAVE_PARENT_SHA` (PW-4) | BLOCK + re-cut |
| **PT-3** | Worktree path exists + clean | BLOCK + recreate |
| **PT-4** | AgentRunner config carries worktree + branch + file_ownership + parent_branch + wave_siblings + pre_dispatch_sha | BLOCK + reject as incomplete |
| **PT-5** | PermissionMatrix fail-closed (Cookbook §65) | BLOCK with `BuilderError::MissingPermissionMatrix` (CWE-276) |
| **PT-6** | W3C_TRACEPARENT env var set (observability-canon §1.1) | BLOCK + regenerate |
| **PT-7** | **Cookbook §64.8 Git-Context Preamble injected** into AgentRunner system prompt | **BLOCK — non-negotiable; workers without preamble lose git-awareness** |

#### Post-Task (3 gates, fires per task AFTER `IMPLEMENTATION_COMPLETE`)

| Gate | Check | Failure → |
|------|-------|-----------|
| **PoT-1** | Worker stayed in scope: `git diff --name-only HEAD~1` ⊆ task.file_ownership | BLOCK merge + REVIEW_FAIL → FixAgent |
| **PoT-2** | Tree-hash matches worker report (§15.4.5 phantom commit prevention) | BLOCK + diagnose |
| **PoT-3** | decisions.md entry written with `manifest_id` + active HKDF subkey-id + task_id | BLOCK + diagnose append failure |

#### Post-Wave (3 gates, fires per wave AFTER all tasks merge)

| Gate | Check | Failure → |
|------|-------|-----------|
| **PoW-1** | All Wn task branches merged to feat/ (no orphans) | BLOCK Wn+1; HITL |
| **PoW-2** | Worktree count = 1 main + 0 task (§64.7 + Ironclaw §9 max-8) | BLOCK + cleanup |
| **PoW-3** | `git ls-tree -r feat/{build}` contains all Wn merge artifacts (§15.4.5 containment) | BLOCK + diagnose missing |

#### Total: 4 + 7 + 7 + 3 + 3 = **24 git-aware gates** per wave-lifecycle

(Note: Cookbook §64.8 Git-Context Preamble injection at PT-7 is the 25th explicit invariant; §SG-CRYPTO.1 manifest verification + §SG-CRYPTO.2 subkey rotation + §SG-CRYPTO.3 hash-chain integrity per gate = 28 total when counting cross-doc enforcement.)

### §15.3.13.6 Wave dispatch state machine

```
Wave-N: pending → preflight → cutting → dispatching → executing → reviewing → merging → complete
                                                                                          → fail (≤3 retries) → escalated
```

State transitions emit AYIN spans:
- `wave.cut { wave_id, parallelism, feat_head_sha }`
- `wave.dispatch_start { wave_id, task_count }`
- `wave.execute { wave_id, in_flight_count }` (gauge, sampled 1Hz)
- `wave.review_complete { wave_id, verdict, fix_iter_count }`
- `wave.merge_complete { wave_id, merge_sha }`
- `wave.escalated { wave_id, reason, l4_escalation_id }`

## §7.8 — Pre-Completion Verification Gate (2026-05-18 ADDITION, candidate #16)

**Scope**: Operationalizes the `⚡PRE-DONE` marker pattern from ironclaw-spine iter-11. When a plan documents tasks completed inline during plan authoring (self-referential pre-execution), /BUILD MUST verify-not-execute rather than re-running the work.

### §7.8.1 Marker syntax

In a LASDLC v2.5.2 wave decomposition table, pre-completed tasks carry `⚡PRE-DONE` suffix in the Task ID column with attribution:

```markdown
| p2-w1-r1-prior-art ⚡PRE-DONE (TR-1) | task/{build}/p2-w1-r1 | R1 research findings | researcher |
```

A `> **⚡ PRE-COMPLETED [date] (TR-N)**: <narrative + /BUILD action guidance>` callout above the wave table documents what was done and what /BUILD must verify.

### §7.8.2 Verification protocol

When /BUILD reaches a wave containing ⚡PRE-DONE tasks, instead of dispatching AgentRunner instances:

1. **Staleness check** (Canon XXXV ≤14 days): compute days since pre-completion stamp; if >14 days → re-execute as fresh task, NOT verify
2. **Artifact existence check**: verify the file/section/code referenced in the pre-completion narrative still exists
3. **Canon-drift check**: SHA256 of any canon docs cited by the pre-completed work; compare against SHA at pre-completion time; mismatch → flag for canon-amendment review before flip
4. **Operator HITL on stale verification**: if ≥2 of {staleness, artifact, canon-drift} fail → §HITL-7 escalation; operator decides re-execute vs accept stale

If all 3 checks PASS: flip status `pre_done → verified_complete`; emit `wave.pretask_verified { task_id, pre_done_age_days }` AYIN span; proceed to next wave.

### §7.8.3 Pre-completion provenance requirement

Pre-completion markers MUST cite:
- Session date when pre-completion occurred
- Operator-authorization signal (Canon XV stamp, /XEA verdict reference, or explicit instruction)
- Cross-reference to where the artifact lives (plan §, canon section, file path)

Without provenance, pre-completion is unverifiable and reverts to fresh execution.

### §7.8.4 Anti-pattern — silent pre-completion

Tasks completed during plan authoring without ⚡PRE-DONE marker = silent pre-completion. /BUILD has no way to distinguish "already done" from "needs execution" and will re-execute, producing duplicate work or canon drift. **Mark explicitly or expect re-execution.**

## §7.9 — Implementation-Readiness Audit (2026-05-18 ADDITION, candidate #13 formalization)

**Scope**: A distinct post-/PLAN review class that catches concreteness gaps no quality/security/perf/a11y review surfaces. Runs AFTER SCRUM R3 returns VALIDATED, BEFORE /BUILD fires.

### §7.9.1 What it catches (12-dimension audit)

The /BUILD executor opens the plan tomorrow with no planning-session context. It needs the literal bytes to execute. Audit the plan body for:

1. **Type signatures** — every new type has full schema (Rust + TypeScript)
2. **Function signatures** — every NEW function in the File-Function Map has a complete signature in the plan body
3. **File paths** — every path is absolute and resolvable on the target machine
4. **Test commands** — every test command in exit criteria is copy-pasteable
5. **Existing patterns referenced** — verify against pinned SHA they actually exist
6. **External APIs** — endpoint paths, auth headers, response shapes pinned
7. **Algorithm choices** — every algorithmic primitive (hash, cipher, codec) has a declared dep
8. **Cross-references valid** — `file:line` refs against pinned SHA
9. **Edge cases** — empty/error/loading states for every new view
10. **Dependencies** — every new lib named + version-pinning strategy
11. **Acceptance criteria checkable** — objective pass/fail tests, not "feels right"
12. **Phase ordering hazards** — Phase N depends on Phase M's output, explicit handoff

### §7.9.2 Severity scale

- **BLOCKER** — cannot proceed without resolving (e.g., undeclared dep, missing struct schema)
- **STUCK** — must ask a clarifying question before continuing (e.g., "which library version?", "where does this state live?")
- **SLOW** — can proceed but slowly (e.g., must grep around codebase for existing patterns)

### §7.9.3 Fold-trigger

Treat ≥1 BLOCKER as iter-N+1 fold-in trigger. Track in plan verdict block as a SEPARATE fold-iteration distinct from SCRUM rounds.

### §7.9.4 Why distinct from SCRUM

SCRUM R3 returning VALIDATED converges same-author multi-iteration review to ~93 STRONG band. Cannot reach EXEMPLARY without independent external implementation-readiness audit pushing the ceiling another +0.5 to +1.0 aggregate. The /BUILD agent does not have access to the planning session's context — only what's literally in the plan body. SCRUM cross-sibling lenses catch *design* gaps; implementation-readiness audit catches *concreteness* gaps. Both are needed; neither substitutes for the other.

### §7.9.5 Pressure-tested

`gitforest-live-ops` iter-7 audit surfaced 12 BLOCKERS + 11 STUCK + 8 SLOW that 6 prior iterations missed entirely. iter-7 fix pass closed all 12 BLOCKERS verified by sanity audit (23/23 RESOLVED). Without this dedicated audit class, those gaps would have surfaced at /BUILD execution time as halt-and-clarify events, much more expensive to resolve.

### §7.9.6 Composition

- Runs after Step 5.2 (SCRUM R3 verdict) per /PLAN skill
- Auditor: independent agent (cold-context Explore or different sibling) per Canon XXXIII independent-runner principle
- Output: severity-classified findings list appended to plan verdict block as `implementation_readiness_audit:` sub-block
- Iteration: ≥1 BLOCKER triggers iter-N+1 fold; re-audit on iter-N+1 output

## §Phase-2A.5 — Canon-Doc Amendment Phase Protocol (2026-05-18 ADDITION, candidate #17)

**Scope**: When a build plan introduces patterns that contradict, extend, or fill gaps in existing canon docs, the plan SHOULD insert a dedicated canon-doc amendment phase between research and implementation phases. This closes the **canon-violation-by-construction** risk where implementation lands code that contradicts canon-doc declarative statements.

### §Phase-2A.5.1 Trigger conditions

A plan triggers this protocol when ANY of:

1. **Declarative contradiction**: implementation introduces capability that an existing canon section declares absent (e.g., webshell-api-surface §2.10 declared "no worktree operations exist" but plan implements them)
2. **Schema extension**: plan extends a canon schema (e.g., LASDLC v2.5.1 → v2.5.2 wave schema addition)
3. **New canon doc required**: plan introduces a discipline that lacks any canonical anchor (e.g., observability-canon.md as 9th doc for AYIN gates)
4. **Cross-canon dependency**: plan touches ≥3 canon docs in coordinated ways (security + observability + agents-playbook + cookbook)

### §Phase-2A.5.2 Phase structure

Canon-amendment phase sits BETWEEN the security/observability foundation phase (Phase 2A in ironclaw pattern) and the spine-implementation phase (Phase 3). Typical contents (parallel-safe, single wave):

| Task class | What | Who |
|---|---|---|
| Doc-edit | additive canon section per affected doc | knowledge |
| Drift-fix | correct factually-wrong prose in existing canon | knowledge |
| New-doc | author NEW canon doc if needed | knowledge + domain owner |
| .md/.html sync | for paired docs (webshell-api-surface), bump both siblings | knowledge |
| Property test | `tests/canon_doc_integrity.rs` enforces post-edit invariants | testing |
| LÆX queue file | submit Canon XXXIX 4-step pipeline candidate per amendment | LÆX |

### §Phase-2A.5.3 Gate criteria

Phase passes only when:

1. All canon-doc edits committed with `<!-- IRONCLAW-{build} CANON AMENDMENT -->` attribution headers
2. `tests/canon_doc_integrity.rs` PASSES (count match, no orphan references)
3. .md/.html sibling sync verified (HTML 4-location bump per webshell-api-surface §6.6)
4. LÆX Canon XXXIX 4-step pipeline queue entries filed for each amendment
5. Operator stamp (Canon XV override) recorded OR LÆX ratification deferred to Phase 7

### §Phase-2A.5.4 Why this phase exists

**Anti-pattern**: implementing capability without amending the canon section that contradicts it = the implementation IS canon-violating from day 1. Even when LÆX eventually ratifies the amendment at Phase 7, the intervening weeks have code-in-flight that contradicts canon. Auditing this state is impossible — operator can't tell whether canon or code is authoritative.

**This protocol** lands canon edits BEFORE implementation begins, so during phases 3-6 the canon doc and code are mutually consistent. Phase 7 LÆX ratification then formalizes what's been operationally true since Phase 2A.5.

### §Phase-2A.5.5 Pattern source

ironclaw-spine §5.7 + §5.8 (canonical instance, 2026-05-18 authoring session). See ironclaw-spine plan for the reference exemplar.

## §7.10 — Two-Problems-in-One-Question Framing Discipline (2026-05-18 ADDITION, Phase 7 ratified, candidate #22)

**Scope**: Pre-implementation triage discipline. Applies when operator requests bundle two solutions in one sentence ("we need X or maybe Y").

### §7.10.1 The problem

Operator requests that bundle two solutions ("do X or do Y") often hide two distinct concerns with different urgency and cost profiles. Conflating them produces over-engineered solutions or wrong-problem recommendations.

### §7.10.2 The framing test

When you see "do X or do Y" language:

1. **Restate as two separate questions**: "X solves what?" + "Y solves what?"
2. **Check problem distinctness**: do X and Y solve different problems or different aspects of the same problem?
3. **Match problems to symptoms**: which problem matches the immediate trigger for the request?
4. **Tier independently**:
   - Problem 1: urgent cost/benefit → Tier 1 (execute now) or conditional
   - Problem 2: conditional cost/benefit → Tier 2 (defer until trigger condition) or orthogonal
5. **Present trade-offs per-problem**, not per-solution

### §7.10.3 Why this catches real gaps

Bundled phrasing masks diagnostic uncertainty. The honest read is often "I noticed something wrong and have multiple hypotheses." Treating it as "pick X or Y" produces recommendations that solve one problem confidently while ignoring the other, or solve a compound problem when only one was urgent.

### §7.10.4 Pressure-tested

2026-05-18 ironclaw-spine iter-18. Operator request: *"move the changelog somewhere else **or** create a repo for canon files."* Framing test surfaced two problems:

| Problem | Symptom | Solution | Cost | Recommendation |
|---|---|---|---|---|
| Schema-bloat (changelog accretion inside YAML) | LASDLC at 7567 lines, v2.5.4 about to be 4th tail block | Extract to `<doc>.CHANGELOG.md` companion | 10 min | Tier 1 — DO NOW |
| Canon access-control / publication-readiness | Canon lives in sdk repo; sdk may go public | Move canon to dedicated repo | Multi-day; reverses recent migration | Tier 2 — DEFER until trigger |

The "OR" framing made these look like alternatives; they weren't. Applied Tier 1 in 5 commits (30 min); Tier 2 deferred with explicit trigger conditions. Conflating them would have produced an over-engineered "new canon repo with CHANGELOG separation built in" project addressing the wrong problem (sdk private, no publication need) while leaving the actual symptom unsolved.

### §7.10.5 Composition

- Canon XV (Principal Hierarchy — operator authority over problem diagnosis)
- Architects Blueprint Part I (Architectural Thesis discipline — thesis clarity prerequisite)
- Builders Cookbook §66 (Context Assembly Discipline — context is problem-specific)
- agents-playbook §7.9 (Implementation-Readiness Audit — catches similar framing gaps at concreteness level)

### §7.10.6 Anti-pattern

Solving the wrong problem confidently. The framing discipline is the gate that catches it before commitment.
