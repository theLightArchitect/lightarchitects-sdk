§0 — For New Readers

The webshell is a local web application that turns your browser into a full engineering interface for the Light Architects platform. Start it with lightarchitects webshell start, open http://localhost:8733, and everything — running agent sessions, managing builds, browsing the knowledge graph — happens through the UI. No terminal fallback required.

Two binaries, one port — lightarchitects-webshell serves the SPA and all /api/* routes documented here. A separate lightarchitects-gateway binary handles the cloud platform API (/v1/*). Different ports, different scopes. This document covers the webshell only.

Authentication

Auth Mechanisms

Three distinct credential paths are supported. All token comparisons are constant-time via constant_time_eq to prevent timing side-channel attacks.

MechanismEndpointToken locationUse case
Bearer headerAny authenticated routeAuthorization: Bearer <token>CLI clients, initial SPA load before exchange
Session cookieAll same-origin requests after exchangeCookie: la_session=<token> — HttpOnly; SameSite=Strict; Secure; Max-Age=28800SPA post-exchange (8-hour sliding TTL)
WS sub-protocol/api/terminal/ws, /api/builds/{id}/terminal/wsSec-WebSocket-Protocol: bearer.<token>PTY terminal — browsers cannot set Authorization on new WebSocket()
Notify tokenPOST /api/builds/{id}/notifyX-LA-Notify-Token: <64-char hex>Per-build 32-byte secret; consumed by external notifiers

Why Two Auth Surfaces?

Browser vs. machine callbacks — never interchangeable.

The session cookie (la_session) is HttpOnly, preventing browser JavaScript from ever reading it — XSS cannot steal it. SameSite=Strict prevents CSRF.

The notify bearer token (Config::token) is used exclusively for machine-to-machine callbacks — agent processes notifying the server of gate results or events. It is deliberately excluded from BuildResponse (the JSON the browser receives after a build is created), so browser JavaScript can never access it.

If the notify token were readable by browser JS, an XSS vulnerability could exfiltrate it and forge agent callbacks — turning a UI-layer exploit into server-side event injection. The two surfaces are separated in code, not just by convention. This is a CWE-306 prevention pattern: see src/agent/bridge.rs for the omission point.

§2.33 MCP Host Proxy (webshell-mcp-host — 2026-05-21 POST-MERGE BACKFILL)

Generic stdio MCP server proxy. The webshell can host any MCP server declared in ~/.lightarchitects/webshell-mcp.json and proxy tool invocations through a scope + JSON-Schema gate.

5-layer trust model: env isolation → sandbox-exec → process group → ScopeGovernor+SchemaValidator → TOCTOU-safe check. All routes require AuthGuard (cookie session, not bearer). If no MCP host is configured, all three return 503.

§2.34–§2.35 Project Registry (webshell-project-ingestion — 2026-05-21)

Two endpoints for registering and reading projects in the local project registry at ~/.lightarchitects/projects/. All routes require AuthGuard (cookie session).

MethodPathAuthPurpose
GET/api/projects/:slugcookieRead project metadata; 404 MANIFEST_MISSING if not registered
POST/api/projects/initcookieRegister project; write TOML manifest (O_CREAT|O_EXCL atomic)
Security properties: Slug validation rejects path separators and empty strings at route entry. POST /api/projects/init uses O_CREAT | O_EXCL for atomic manifest creation — no TOCTOU race. Glossary

Quick definitions for terms used throughout this document without prior introduction.

TermDefinition
PTY Pseudo-terminal — a kernel-level pair (master + slave) where the webshell server holds the master end and the agent process holds the slave end. Makes the agent believe it is running in an interactive terminal, enabling streaming I/O and TTY control sequences.
AgentSession Tagged serde enum (src/config.rs:244, #[serde(tag = "agent")]) that carries backend-specific config. The agent field in the config file selects the variant: lightarchitects, codex, lightarchitects_native, or mistral_vibe.
vibe-acp The MistralVibe agent binary. Communicates over stdin/stdout using the Agent Communication Protocol (ACP). MISTRAL_API_KEY is injected by spawn_bridge at fork time — never present in the parent process environment.
SOUL helix The platform's long-term knowledge graph: enriched session memory, architectural decisions, and domain knowledge stored as vector-embedded entries in a Neo4j-backed graph. Browsable via the /helix screen (/api/soul/* routes).
ScreenKey TypeScript union type (src/lib/routes.ts:5) naming the 11 navigable screens. The hash router maps URL hash patterns to a ScreenKey; app.svelte lazy-loads the matching Svelte component via screenModules.
spawn_bridge Rust function (src/agent/bridge.rs) that forks the agent process: calls env_clear() to strip the parent environment, re-injects a whitelisted set of env vars, builds binary-specific args, and wires PTY I/O. All agent sessions start here.
la_session HttpOnly session cookie (Max-Age=28800, SameSite=Strict, Secure, Path=/). Issued at /api/auth/nonce-exchange; consumed by AuthGuard middleware on every browser-facing route. Never readable by browser JavaScript.
X-LA-Notify-Token Convention name for the static bearer token used in machine-to-machine callbacks. The actual value is Config::token. Excluded from BuildResponse by design — see the Why Two Auth Surfaces callout for the security rationale.
