# Path A — Decisions Record

Resolutions for the 8 open issues raised in:
- `arch/cockpit-d2-manifest-extension.proposal.yaml` (M-1 … M-5)
- `arch/cockpit-d2-fleet-extension.proposal.md` (F-1 … F-3)

Decisions taken by Claude on 2026-06-05 to unblock implementation. **Operator stamp required** before merge to main; decisions reflect best judgment + Cookbook/Canon discipline. Each line answers: *what was chosen, why, and what changes downstream.*

---

## M-1: `status` field unification

**Decision**: Split the pipe-separated `status: PROMOTED | MERGED` into two fields:
- `plan_status`: enum `[draft, promoted, abandoned]` — planning lifecycle
- `build_status`: enum `[pending, active, complete, failed]` — execution lifecycle

**Why**: The pipe-form is ambiguous (history? AND? OR?). The two phases of a build (planning vs execution) are orthogonal lifecycles and deserve independent representation.

**Downstream**: LASDLC template v2.9.0 enforces the structured form. Existing manifests with pipe-form: handler parses both legacy field name as `plan_status="promoted" + build_status` inferred from `merged_sha` presence (M-5 degraded mode).

---

## M-2: Amendment migration

**Decision**: New builds (post-contract-ratification) populate `phases[].amendments[]` directly via `/BUILD` wave-dispatcher when a wave surfaces a scope pivot. Existing builds' plan.md prose remains as-is; a separate `/SYNC --extract-amendments` tool will backfill in a future iteration.

**Why**: Don't block on backfill of ~50 historical manifests. The cockpit's value is for active builds; historical reads can use degraded mode (empty amendments array).

**Downstream**: `/BUILD` skill needs a new sub-action `record_amendment(wave, desc, impact)` that appends to manifest.yaml under `phases[$current].amendments[]`.

---

## M-3: Commit SHA writeback

**Decision**: `/BUILD` wave-dispatcher writes `phases[i].waves[j].commits[]` after each commit it produces. Hook point: `lightarchitects/src/lightsquad/wave_dispatcher.rs` on commit completion event.

**Why**: Explicit mapping beats commit-message inference. Inference is brittle (wave IDs not always in messages, history rewrites break it).

**Downstream**: For v0.1.0 the gateway handler reads only what's present. Manifests without `commits[]` per wave return empty arrays — the cockpit shows the wave with no SHA pill but otherwise renders normally.

---

## M-4: Task ID generation

**Decision**: Task IDs auto-generated by `/PLAN` finalize step as `t<phase>.<wave>.<n>` (e.g. `t3.2.1`). Plan markdown carries `<!-- id: t3.2.1 -->` HTML comments next to each task bullet so they survive operator edits.

**Why**: Operator-friendly (read/edit the plan as markdown; the IDs are just metadata). Survives renames if the comment stays attached to the bullet.

**Downstream**: `/PLAN` skill needs a finalize sub-step that injects IDs. `/BUILD` reads IDs from the comments to populate manifest.yaml `phases[].waves[].tasks[].id`.

---

## M-5: Backward compatibility (degraded mode)

**Decision**: Handler falls back gracefully when manifest lacks new fields.

| Missing field | Fallback behavior |
|---|---|
| `phases[]` (only integer count present) | Synthesize a single phase `{id: "p0", name: "Unstructured", status: derived-from-merged_sha}` |
| `phases[].waves[]` | Empty array |
| `phases[].waves[].commits[]` | Empty array, or single entry built from `merged_sha` + `merge_commit` if at build level |
| `phases[].waves[].tasks[]` | Empty array |
| `phases[].amendments[]` | Empty array |
| `started_at` | Use file mtime of plan.md as proxy |

**Why**: ~50 historical manifests exist in `active.yaml`. Forcing migration before any read works is wrong. Degraded mode lets the cockpit render them as "this is a legacy build, here's what we have."

**Downstream**: Gateway handler implementation must include `manifest_v2_or_legacy()` parsing logic that tries the new shape and falls back. Conformance test fixture needs both a v2 manifest and a v1-style manifest.

## F-1: `focus_target_fn` shape

**Decision**: String for v0.1.0. Refactor to `{kind, name, file}` in v0.2.0.

**Why**: String is enough for the cockpit's display today (renders as "fn handle_message"). Structured form is needed when d3 starts using it to scroll to the symbol — that's v0.2.0 work alongside symbol-drill plumbing.

**Downstream**: `FleetSpan.focus_target_fn: Option<String>` in v0.1.0. The string convention is `<kind> <name>` (e.g. `"fn handle_message"`, `"struct AgentSession"`). v0.2.0 introduces a parser + struct.

---

## F-2: Same-span mutation on task transition

**Decision**: Same `FleetSpan`, mutable `task_id`. Add method `FleetTracker::agent_focused_on(agent_id, new_task_id)`.

**Why**: Agent identity is stable across tasks within a wave. Creating a new span per task explodes span count + breaks AYIN time-aggregation. The agent isn't a *new* agent when it switches tasks — its focus shifts.

**Downstream**: `agent_focused_on` mutates `task_id` and `focus_target_fn`. The `running_ms` resets to 0 on task switch (it's per-task focus, not span lifetime).

---

## F-3: Idle clearing

**Decision**: `task_id = None` when no task is active. No sentinel value.

**Why**: `Some("idle")` is a typed lie — it makes idle look like a task in every downstream consumer (cockpit, AYIN aggregations, contracts). `None` is the honest signal that maps cleanly to JSON `null` and to SQL `NULL` if persistence enters the picture.

**Downstream**: Cockpit treats `agents[].focus_task_id == null` as "agent is wave-bound but not task-focused" and surfaces it under the wave node, not under any task row. The agent badge in the wave row is dim (not `running` cls).

## What this unblocks

With these eight decisions in place:

- Contract YAML can move to canonical location (`standards/canon/contracts/wire.http/`).
- Gateway handler implementation can start (Phase 3).
- FleetNode/FleetSpan extensions can land (Phase 4).
- v0.2.0 scope is now sharply defined: SSE streaming endpoint, symbol-aware `focus_target_fn`, `/SYNC --extract-amendments` tool, `/PLAN` task-ID injection step.
